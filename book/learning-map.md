# 学习地图

## 先学什么

对已经做过 dApp 的前端来说，最重要的问题不是“还能学什么”，而是“先学什么最值”。因为 Web3 的知识面太大了，大到你如果没有顺序，几乎一定会陷入一种熟悉的状态：今天看 EVM，明天看黄皮书，后天看安全文章，大后天看 Uniswap 源码，结果每样都碰了一点，但没有一块真正长成主干能力。

所以这本书的第一原则是：先建立主干，再扩展分支。

对你的阶段来说，最有价值的学习顺序通常是：

1. Solidity 与状态模型
2. 交易执行链路
3. EVM 与 Gas
4. 安全
5. 工程化与测试
6. 协议源码阅读
7. 项目实战

这个顺序背后的逻辑不是“由浅入深”这么简单，而是“先补最影响工程判断的认知”。

为什么先不是 EVM？

因为如果你还没有真正理解状态设计和合约接口，直接看 EVM 往往会变成记术语。你会知道有 stack、memory、storage、opcode，但这些概念并不会自动转化成写代码和看代码时的判断。

为什么安全也不是第一章？

因为安全不是孤立知识。你只有先理解状态、调用、权限和失败路径，安全问题才会有真实的落点。否则漏洞名词会像考试题，而不是工程风险。

为什么源码阅读要放在后面？

因为源码阅读的真正价值是“把前面的模型在真实系统里对上”。如果你前面的模型还没建立起来，那么读源码时你往往只能看懂局部，看不清系统。

## 不要一开始就钻进去的内容

很多前端开发者在想深入 Web3 时，会被两种焦虑推动：

第一种焦虑是“我是不是底层一点都不懂”。  
第二种焦虑是“别人都在看很硬核的东西，我是不是应该也赶紧去看”。

这两种焦虑很容易把你带去错误的第一站。比如：

- 黄皮书
- 共识算法实现细节
- 状态树和 trie 的严密推导
- 零知识证明和密码学证明过程
- 桥和跨链消息验证机制
- 各种高级 MEV 和 L2 证明系统

这些内容有价值，但问题在于：它们不应该挡在你的主干能力前面。

如果你现在最常见的困惑仍然是下面这些：

- 为什么某个写法 gas 更高
- 为什么一个合约会有重入风险
- 为什么一个 event 不能等于链上状态
- 为什么 `delegatecall` 这么危险
- 为什么 upgrade 合约那么多约束

那你需要的不是先去补最深理论，而是先把和这些工程问题直接相关的那部分底层理解打通。

换句话说，不要把“更底层”误认为“当前更该学”。  
你需要先补的是“和你当前工程能力最耦合的底层”。

## 三个月主干路线

如果你每天能稳定投入 2 到 3 小时，下面这条三个月路线是很现实的。

### 第一个月

第一个月只做两件大事：吃透 Solidity 常用能力，和建立状态思维。

重点目标：

- 真正理解 `storage`、`memory`、`calldata`
- 理解 `mapping`、`struct`、`event`、`error`、`modifier`
- 能独立写出 ERC20、简单金库、投票系统
- 开始用 Foundry 写基础测试

这个月最重要的不是“写很多功能”，而是每写一个功能都能回答：

- 状态放在哪
- 谁能改
- 谁能读
- 失败时怎么回滚和报错

如果第一个月做得扎实，你会第一次感觉合约代码不再只是“另一个语法的后端”，而是一个受状态成本和执行环境约束的系统。

### 第二个月

第二个月的任务是往下钻到执行模型和风险模型。

重点目标：

- 理解 stack、memory、storage、calldata 的角色分工
- 看懂 storage slot 和 packing 的实际意义
- 通过实验理解 gas 差异，而不是只背结论
- 系统补重入、权限、签名、代理、预言机等安全问题
- 把 fuzz、fork test、trace 和 gas report 接进日常开发

这个阶段的突破点在于：你开始不再只看“这段代码能不能跑”，而是开始看“这段代码为什么这样跑、为什么要这样写、为什么这里危险”。

### 第三个月

第三个月开始把前面的模型压进真实系统。

重点目标：

- 阅读 OpenZeppelin 的标准实现
- 阅读 Uniswap V2 的关键合约
- 初步理解 Aave 这类更复杂协议的组织方式
- 设计并实现自己的 MultiSig、Staking 或小型 DEX
- 把“会写”进一步提升到“知道为什么这样写”

这个月你最容易获得的提升，不是多学一个知识点，而是把一堆分散知识第一次整合成系统判断。

## 怎么判断自己真的学会了

很多工程师会用一个很模糊的标准判断自己有没有学会，比如“我大概看懂了”“我应该知道这是怎么回事”“我做过类似项目”。但这些判断很容易高估自己。

更好的判断方式，是看你能不能完成下面这些具体动作：

### 第一层：能解释

- 能解释一笔交易从签名到 receipt 的完整流程
- 能解释 `view` 调用和真实交易的差别
- 能解释为什么某些状态写入成本更高
- 能解释 event、receipt、state change 分别是什么

### 第二层：能设计

- 能独立设计一个中小型合约的状态结构
- 能写出合理的权限模型
- 能明确列出失败路径而不是只写 happy path
- 能决定什么时候该用标准实现，什么时候该自己写

### 第三层：能审视

- 能发现最基础的一些安全问题
- 能对 gas 优化保持节制，而不是盲目追求技巧
- 能读懂标准实现背后的设计意图
- 能看源码时抓住“状态、入口、边界、事件、资金流”这几个核心问题

如果你能稳定做到这些，说明你已经从“知道怎么调”进入“知道系统为什么这样运作”了。

## 这条路线为什么适合前端开发者

前端开发者在 Web3 里有一个天然优势：你其实非常接近真实用户操作和真实系统入口。你最容易接触到：

- 用户签名
- 钱包交互
- 交易广播
- pending / success / fail 状态
- event 驱动的 UI 更新
- 链上数据和链下数据的拼接

这意味着你并不是从零开始理解区块链系统，而是已经站在系统边界上了。

但前端开发者也有一个常见盲点：由于平时接触更多的是“消费接口”，很容易把合约看成一个 RPC 终点，而忽略它作为状态机、执行环境和安全边界的本质。

所以你的学习路线，不该照搬纯后端、纯安全研究员或纯协议工程师的第一阶段路线。你最适合的是：

- 先从交易和状态模型补洞
- 再往下钻执行模型
- 再把这些理解带回工程、安全和源码

这条路不仅更高效，也更符合你原本的优势。

- 给自己写一个 30 天计划：每周一个主题，每周至少一个最小合约练习和一组测试。
- 按“状态设计 -> 权限模型 -> 失败路径 -> 测试清单”的格式，为接下来要做的一个项目先写设计稿。
