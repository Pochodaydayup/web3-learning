# Part 4: 安全

## 先理解什么

Web3 和很多普通应用开发最大的不同之一，不是技术栈不同，而是错误代价不同。

在传统前端或普通 Web 应用里，一个 bug 很常见的后果是：

- 页面出错
- 某个功能不可用
- 数据展示错乱
- 用户流程中断

这些问题当然也很严重，但通常还留有回滚、修复、热更新或数据补救空间。

而在智能合约世界里，很多错误的后果是：

- 资金被拿走
- 权限被劫持
- 状态被永久污染
- 协议被恶意操纵
- 修复窗口极小甚至不存在

所以安全在这里不是“上线前加一道检查”，也不是“等审计公司来告诉我哪里不对”。  
对合约工程来说，安全必须是设计的一部分。

要理解这一点，你需要先把安全问题从“漏洞名词表”变回“系统边界问题”。

大多数高频安全问题，本质上都和下面这些问题有关：

- 谁能调用？
- 谁能改状态？
- 调用顺序是否安全？
- 外部依赖是否可信？
- 某个输入能否被伪造、重放或操纵？
- 当外部调用返回后，我之前的假设还成立吗？

只要你从这个角度看，所谓安全就不再是神秘学，而是工程判断能力的自然延伸。

### 先抓住最常见的风险面

这一章不会试图把所有链上攻击一次讲完。  
第一版先抓最关键、最常见、也最值得工程师优先形成肌肉记忆的几类问题：

- 重入
- 权限控制错误
- 签名重放
- `delegatecall` 风险
- `tx.origin` 误用
- 预言机操纵
- Flash Loan 辅助攻击

它们之所以重要，不只是因为“考试常见”，而是因为它们会不断出现在真实系统里。

## 为什么重要

### 合约不是“写对逻辑”就够了

很多工程师刚转向合约开发时，会不自觉沿用普通应用的开发习惯：

- 功能先跑通
- 边界后补
- 先看 happy path
- 上线前再整体检查

在 Web3 里，这种顺序非常危险。

因为攻击者不会像普通用户那样使用你的系统。  
他们会主动寻找：

- 状态更新顺序的漏洞
- 权限判断的遗漏
- 外部调用后的不一致
- 可重复使用的签名
- 可操纵的价格输入
- 可被放大的边界条件

也就是说，安全问题并不是“写错了某一行”，而往往是“你对系统边界的假设太乐观了”。

### 一次错误可能不是重跑部署能解决

前端世界里，你习惯了热更新、回滚、灰度、快速修复。  
合约世界里，一旦错误进入不可控状态，补救往往极其昂贵。

这会直接影响你的工程思维：

- 你不能只测成功路径
- 你不能默认调用者“会按预期来”
- 你不能默认输入来自诚实环境
- 你不能默认状态在一次外部调用前后保持你预期的样子

因此，安全不是额外技能，而是合约系统设计的默认视角。

## 从几个典型问题开始理解风险

### 重入：外部调用之后，世界可能已经变了

重入是最经典的安全问题之一。  
它的核心不是“某个函数写错了”，而是你在外部调用之后，仍然假设当前状态和控制流是安全的。

最典型的错误模式是：

1. 先向外部地址转账或调用
2. 再更新自己的内部状态

如果外部地址是一个合约，它就可能在你状态尚未更新时，再次回调你的函数。  
于是攻击者可以利用这个窗口，反复进入，直到把可提取资源超额拿走。

真正值得你记住的不是“重入这个词”，而是这句判断：

> 只要你把控制权交给了外部世界，外部世界就不再受你的善意假设约束。

这句话几乎能解释大量合约安全问题。

### 权限控制：谁能做什么，必须明确到代码层

很多合约并不是被复杂攻击攻破，而是被最朴素的权限遗漏击穿。

常见问题包括：

- 某个本应管理员调用的函数没有权限保护
- 管理员权限过大且没有分层
- 权限模型只适合现在，不适合未来扩展
- 重要参数可被任意地址更新
- 暂停、升级、提款等高危能力没有明确边界

这也是为什么你必须真正理解 `Ownable` 和 `AccessControl`，而不是只会复制导入。

你需要判断：

- 这个系统是一把总钥匙就够，还是需要角色分层？
- 哪些能力必须单点保护？
- 哪些能力必须多角色分离？

权限设计不是格式问题，而是系统治理问题。

### 签名重放：签名不是“只要验过就安全”

很多应用会使用离线签名完成授权、白名单、元交易、订单确认等流程。  
这时一个常见误区是：

> 只要恢复出签名人地址，签名就可信。

其实远远不够。

你还必须考虑：

- 这份签名属于哪条链？
- 它是不是只能用一次？
- 它有没有过期时间？
- 它绑定的是哪个合约、哪个动作、哪个上下文？

如果这些边界没有纳入签名内容，攻击者就可能把一份原本合法的签名：

- 在别的上下文里重放
- 在别的链上复用
- 在更晚时间反复使用

所以签名安全的关键不是“会验签”，而是“知道签名到底在承诺什么边界”。

### `delegatecall`：借用代码，保留上下文

`delegatecall` 很强大，也很危险。  
它会执行别人的代码，但保留当前合约的上下文：

- 当前 storage
- 当前 `msg.sender`
- 当前地址语义

这使得它非常适合代理升级模式，但也让风险成倍放大。  
因为一旦被委托执行的逻辑不安全，你损失的不是别人的存储，而是你自己的系统状态。

所以只要看到 `delegatecall`，你就应该立刻进入高警惕状态：

- 它委托给谁？
- 目标地址怎么被设置？
- 初始化是否可重复？
- 存储布局是否一致？

很多升级相关事故，本质上都和这些问题有关。

### 预言机与 Flash Loan：你的“外部真实世界”可能是假的

很多协议需要价格数据、资产估值、兑换比例或外部状态输入。  
这些输入一旦可被操纵，整个协议逻辑都可能在“看似正确的内部执行”下被击穿。

Flash Loan 之所以危险，往往不是因为借钱本身，而是因为它让攻击者能在单笔交易内临时拥有极大资金体量，从而：

- 扭曲池子价格
- 放大套利
- 瞬时影响依赖价格的业务逻辑

如果你的协议逻辑默认“价格输入大致可信”，但没有防御机制，那攻击者会把这种假设当成入口。

## 常见误区

### 误区一：以为用了 OpenZeppelin 就自动安全

OpenZeppelin 能帮你避免大量重复造轮子，也封装了很多成熟边界。  
但它不能替你决定：

- 你的状态设计是否合理
- 你的权限边界是否正确
- 你的业务逻辑是否可被利用
- 你的外部依赖是否可信

标准库能降低一部分实现风险，  
不能替代你对系统的整体判断。

### 误区二：以为测试全绿就等于安全

很多测试只是证明：

- 正常流程可以跑通
- 预期输入会得到预期输出

但安全问题往往发生在：

- 不符合预期的调用顺序
- 异常输入空间
- 外部合约的恶意行为
- 权限绕过
- 价格或状态瞬时扭曲

所以测试全绿不等于安全。  
没有攻击视角的测试，往往只能说明“理想用户场景是对的”。

### 误区三：把安全理解成“高深黑客技术”

很多真实事故并不神秘，反而非常朴素：

- 权限没加
- 顺序写反
- 签名边界不全
- 初始化没锁
- 升级入口暴露

安全思维不是“变成顶级攻防专家”，而是先学会系统性地怀疑自己的假设。

### 误区四：觉得项目小就没人打

链上攻击的一个现实是：  
很多系统一旦资产可得、路径明确，攻击者不会因为它“小”就放过它。

更何况小项目通常防守更薄弱。  
所以“项目不大、应该没事”并不是风险判断，而是一种自我安慰。

## 怎么练

### 练习一：故意写一个可重入的 Bank

这是最经典、也最值得做的练习。  
你可以：

1. 写一个简单存取款合约
2. 故意把“先转账、后改状态”写进去
3. 再写一个攻击合约去重入它
4. 最后把它修成“先改状态、后外部调用”并加防护

只要你亲手做过一次，重入就不再只是名词。

### 练习二：检查权限边界

给自己一个简单系统，比如金库或投票合约，列出所有关键函数：

- 初始化
- 提现
- 参数修改
- 暂停
- 恢复
- 升级

然后逐个写清楚：

- 谁能调
- 凭什么能调
- 调完影响什么

这个练习能让你开始形成权限图谱，而不是只在函数头上补 modifier。

### 练习三：写一次签名校验

自己设计一个最小签名授权流程，并强制把下面这些都纳入考虑：

- signer
- nonce
- deadline
- chain id
- contract address
- action payload

只要你把这些边界真正思考一遍，后面看到 permit、订单签名、白名单签名都会更清楚。

### 练习四：写安全审查笔记

挑一个最小合约，哪怕只有几十行，也按下面结构做一次审查：

- 核心状态是什么
- 哪些函数能改状态
- 哪些地方有外部调用
- 哪些输入来自不可信来源
- 哪些地方依赖外部价格或外部结果
- 哪些地方需要权限保护

这类笔记会非常直接地提升你的风险敏感度。
