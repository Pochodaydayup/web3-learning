# 第 3 章练习题

## 概念题

1. `stateRoot` 为什么不能简单理解成“所有余额拼起来做一次哈希”？
2. 为什么区块头放 root，而不是直接放完整状态？
3. 账户状态里的 `storageRoot` 在工程上意味着什么？
4. proof 与“直接相信 RPC 返回值”之间的差异是什么？

## 分析题

1. 假设你要向别人证明某个合约的某个 slot 在区块 N 时的值，请说明为什么只给最终结果不够，为什么需要提供 proof。
2. 如果没有树结构，只是一个普通的大对象哈希，会给局部更新和局部证明带来什么问题？
3. 试着分析桥接协议为什么高度依赖 root 与 proof 的组合。

## 代码题

1. 查询一个真实区块的 `stateRoot`，并解释它与该区块之后所有账户状态的关系。
2. 使用支持 `eth_getProof` 的节点，获取一个账户或 slot 的 proof，写出你对返回字段的解释。
3. 画一张“区块头 -> stateRoot -> account proof -> storage proof -> slot value”的关系图。
