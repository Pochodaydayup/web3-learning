# 交易、区块与状态闭环

## 先理解什么

Ethereum 最适合开发者的入门方式，不是从抽象共识算法开始，而是从“一笔交易经历了什么”开始。因为这条链路能把钱包、节点、手续费、区块、EVM 和状态更新都串起来。

你可以把一次典型交易的路径先概括成：

1. 钱包构造并签名交易
2. 节点接收并传播到 mempool
3. 验证者或打包者选择交易进入区块
4. EVM 执行交易并修改状态
5. 生成 receipt、logs 和新的状态根

### 先把几个词钉牢

**Transaction** 是把账户意图和执行参数带进链上系统的状态变更请求。直觉上它像用户递交给链的正式指令单。工程上这意味着交易不是“点了一下按钮”，而是携带 nonce、gas、目标地址和输入数据的一整份执行请求。

**Block** 是按顺序打包一组交易并提交状态结果的区块单位。直觉上它像账本里按批次结算的一页。工程上这意味着单笔交易的命运不只取决于自己，还取决于它被放进了哪个 block、和谁一起被执行。

**State Root** 是某个区块执行完成后全局账户状态的根哈希承诺。直觉上它像把整张世界状态表压缩后的总指纹。工程上这意味着轻客户端、状态证明和跨系统验证，最终都要围绕 state root 建立可信关系。

## 为什么重要

很多表面问题其实都能在这条链路里找到归属：

- 交易卡住，通常和 mempool 排队、手续费或 nonce 有关
- 交易失败，通常发生在执行阶段
- receipt 可见但前端状态不对，往往是链下消费层的问题
- 替换交易、取消交易，本质上和 nonce 与传播规则有关

只要你能把概念放回链路，很多底层知识就不再零散。

## 核心机制

### 1. 账户模型和 nonce 让交易有了顺序

Ethereum 使用账户模型。对外部账户来说，nonce 是每发送一笔交易都会递增的计数器。它的作用很关键：

- 防止同一笔交易被无限重放
- 给同一账户的交易建立顺序
- 支持“同 nonce 替换”这样的行为

这也是为什么你有时会遇到“后发的交易被前面的 pending 卡住”。不是 RPC 抽风，而是账户顺序约束在起作用。

### 2. Mempool 是传播和等待区，不是结果区

交易签名后进入节点，再被传播到 mempool。这里最重要的是：它只是一个待打包区域，不是成功结果区。

在这里，交易可能被：

- 正常传播
- 因手续费过低而不受欢迎
- 因同 nonce 替代而被顶掉
- 因账户条件变化而失效

所以拿到交易哈希之后，你真正拥有的是“一个进入系统的请求”，不是“一个已经稳定落链的结果”。

### 3. 区块把一批交易放进统一执行上下文

当验证者或打包者选择交易进入新区块时，这批交易会在某个具体区块上下文里执行。区块不仅仅是“一个装交易的盒子”，它还携带了执行和排序的时间片语义。

对开发者来说，更实用的理解是：

- 一笔交易只有进入区块，才真正开始链上提交意义上的执行
- 区块顺序决定了状态演化顺序
- 交易不是独立存在的，它总是在某个前置状态之上执行

### 4. 执行结果分成主状态、receipt 和 logs

EVM 执行后，会留下几个层次的结果：

- 主状态变化：真正写进 storage 的长期结果
- receipt：关于这次执行的摘要
- logs：链下系统可消费的事件记录

这三个层次经常被前端混在一起，但你必须分清：

- 业务真相优先看状态
- 执行摘要看 receipt
- 通知和索引常看 logs

### 5. State Root 让全局状态有了可验证摘要

每个区块执行完后，全局状态并不是直接“拷贝了一份全量数据”，而是通过状态树组织，并以 root hash 的形式给出摘要。你不一定要立刻掌握 trie 的实现细节，但至少要知道：state root 的存在，是为了让庞大的全局状态可以被高效校验和引用。

对开发者来说，这层理解的实际意义是：链上状态不是松散变量堆起来的，而是有统一可验证结构的系统。

## 工程判断

学习 Ethereum 底层时，最重要的是避免两种误区。

第一种是完全不学底层，只停留在 SDK 层。这样你会在很多真实问题上判断不稳。

第二种是一上来就钻最深的论文和规范，结果很快失去主线。

更合理的方法是：先抓交易主链路，再逐渐往下扩展。只要你能讲清楚“签名 -> mempool -> 区块 -> 执行 -> receipt -> 状态根”这条路径，后面学 nonce、费用模型、trie、共识时都会更自然。
