# 术语表

这份术语表不是为了让你背定义，而是为了让你在读完全书时，遇到高频词就能立刻知道它在工程里到底意味着什么。你可以把它当成全书的统一词汇索引：哪里读到卡住了，就回来看这个词在账户、执行、协议或工程流程里分别扮演什么角色。

## 账户与交易

- **EOA（外部账户）**：由私钥控制的账户，可以主动发起交易、签名消息、支付 gas。直觉上它像“拿着钥匙的人”。
- **合约账户（Contract Account）**：由链上代码控制的账户，不能自己无中生有地发起交易，只会在别人调用它时执行逻辑。直觉上它像“被触发后才运行的程序”。
- **地址（Address）**：账户或合约在链上的唯一标识。前端里看到的 `0x...`，本质上是在定位一段状态或一份代码。
- **私钥（Private Key）**：控制 EOA 的秘密凭证，谁掌握私钥，谁就能代表该账户签名。
- **公钥（Public Key）**：由私钥推导出的公开身份信息，配合签名算法可以验证“这条签名确实来自某个私钥持有者”。
- **助记词（Mnemonic）**：方便人类保存与恢复钱包的一组单词，本质上是生成私钥种子的可读表达。
- **签名（Signature）**：用私钥对消息或交易做出的加密证明，用来证明“这是我同意发送的”。
- **Nonce**：账户的递增计数器，用来保证交易顺序并防止同一笔授权被重复使用。对前端开发者来说，它直接影响替换交易、并发发送与失败重试。
- **交易（Transaction）**：从账户发往链上的状态变更请求，包含接收方、数据、gas 参数、nonce、签名等信息。
- **消息调用（Message Call）**：合约之间的内部调用。它不是独立交易，但会在某笔交易执行过程中继续触发更多逻辑。
- **Mempool**：节点暂存待打包交易的区域。交易先进入 mempool，再由打包者挑选进区块。
- **Receipt**：交易执行后的结果摘要，常包含状态、gas 使用量、事件日志和区块位置。
- **Gas Limit**：发送者愿意为这笔交易最多提供多少执行资源。它限制的是“最多能跑多远”，不是最终一定花这么多。
- **Base Fee**：EIP-1559 里协议自动调整的基础费用，会被直接销毁，反映区块空间的基础稀缺程度。
- **Priority Fee**：给打包者的额外小费，用来提高交易被优先纳入区块的概率。

## Solidity 与合约结构

- **ABI**：Application Binary Interface，合约对外暴露接口时使用的编码与解码规范。前端之所以能调用合约，本质上是在按 ABI 规则拼装数据。
- **Interface**：只声明函数与事件签名、不写实现的合约接口，用来表达“我预期对方支持哪些能力”。
- **Function Selector**：函数签名哈希后的前 4 个字节，是 EVM 在调用时用来路由到目标函数的入口标识。
- **Modifier**：包裹函数执行前后逻辑的复用结构，常用于权限、暂停、校验等横切规则。
- **Event**：写入日志系统的链上事件，适合给前端、索引器和分析系统消费，但不能被链上逻辑直接当作状态读取。
- **Custom Error**：Solidity 自定义错误类型，用更低成本表达失败原因，比长字符串 `require` 更省 gas。
- **Mapping**：键到值的链上映射结构，适合做权限、余额、状态索引，但不适合链上遍历全部内容。
- **Struct**：把多个字段打包成一个业务对象的数据结构，适合表达订单、仓位、投票等实体。
- **Enum**：有限状态集合，适合表达阶段性状态，如 `Pending`、`Active`、`Closed`。
- **Storage / Memory / Calldata**：Solidity 中最重要的数据位置概念。`storage` 持久化、`memory` 临时可变、`calldata` 外部输入只读。
- **Constructor**：合约部署时只执行一次的初始化逻辑，写入的是部署阶段状态。
- **Initializer**：可升级合约里代替 constructor 的初始化入口，通常需要显式防止重复调用。
- **Inheritance**：通过继承复用父合约状态和逻辑的机制，方便共享能力，但也容易引入复杂的存储布局和调用关系。
- **Composition**：通过拆分模块合约并互相协作来组织系统。相比大量继承，它通常更容易控制职责边界。
- **Proxy**：把状态和逻辑拆开的升级架构。用户地址不变，但背后执行的实现合约可以替换。
- **UUPS**：一种常见的升级模式，把升级入口放在实现合约里，让代理更轻，但要求升级权限设计足够严谨。

## EVM 与执行

- **EVM**：Ethereum Virtual Machine，以太坊执行智能合约的虚拟机。它决定了链上代码能做什么、怎么计费、怎么回滚。
- **Opcode**：EVM 的底层指令，如 `SSTORE`、`CALL`、`JUMPI`。所有 Solidity 代码最终都会被编译成 opcode 序列。
- **Bytecode**：部署到链上的机器码。合约源码只是人类可读形式，真正被执行的是 bytecode。
- **Runtime Bytecode**：合约部署完成后，真正长期存放在地址上的那部分代码。
- **Init Code**：部署时临时运行的创建代码，用来返回最终的 runtime bytecode。
- **Stack**：EVM 的操作栈，保存短生命周期的计算值。它很快，但深度有限，适合做即时运算。
- **Memory**：调用期间的临时可变内存，函数执行结束后就会消失，适合处理中间数据。
- **Storage**：合约永久状态所在区域，写入会进入链上共识，因此最贵、也最关键。
- **Calldata**：外部调用携带的输入数据区域，只读且通常比复制进 memory 更便宜。
- **Returndata**：外部调用返回的数据缓冲区，调用方可以继续读取、解码或透传。
- **Call**：最常见的外部调用方式，在被调合约自己的上下文里执行代码。
- **Staticcall**：只读外部调用，执行期不允许状态修改，适合做查询和安全限制。
- **Delegatecall**：在当前合约的状态上下文里执行别人的代码。它非常强大，也因此非常危险。
- **Revert Data**：调用失败时返回的错误编码，用来告诉上层“为什么回滚了”。
- **Slot**：Storage 的逻辑存储单元。理解 slot，才能真正理解变量布局、packing 和升级风险。
- **Trace**：一笔交易逐步执行时的调用轨迹和 opcode 轨迹，是调试复杂问题时最可靠的现场记录。

## Gas 与性能

- **Gas**：衡量链上执行资源消耗的计费单位。你支付的不是“时间”，而是“协议愿意给你消耗多少计算与存储资源”。
- **Gas Cost**：某个操作实际消耗的 gas 数量。不同 opcode、不同数据位置、不同冷热状态，成本都不一样。
- **SLOAD**：从 storage 读取数据的操作。虽然比写入便宜，但比读 memory 或 calldata 仍然贵得多。
- **SSTORE**：向 storage 写入数据的操作，是最需要谨慎使用的高成本行为之一。
- **Cold Access**：本次交易里第一次访问某个账户或 slot 的状态，成本更高。
- **Warm Access**：同一笔交易里再次访问已经碰过的数据，成本会下降。
- **Packing**：把多个较小变量压进同一个 slot 的布局技巧，用来减少 storage 占用和写入成本。
- **Constant**：编译期固定不变的值，不占用 storage，适合放永远不会改的常量。
- **Immutable**：部署时确定、部署后不可更改的值。它不像 storage 那样长期读写昂贵，又比 constant 更灵活。
- **Refund**：某些状态清理操作在特定规则下可以返还部分 gas，但现代 EVM 对 refund 的激励已经明显收紧。
- **Batching**：把多次相似操作合并成一次执行，以减少固定开销和交互次数。
- **Complexity**：算法复杂度在链上会直接映射成成本和可用性风险，尤其要警惕线性遍历和无上限循环。

## 安全与权限

- **Reentrancy**：外部调用尚未结束时，控制流又回到了原合约，导致同一份旧状态被重复利用的风险。
- **Access Control**：谁可以做什么操作的权限系统。很多事故并不是逻辑不会跑，而是“本不该有权限的人也能跑”。
- **Ownable**：最简单的单管理员权限模型，适合小型系统或明确的单控制角色。
- **Role / AccessControl**：按角色拆分权限的模型，适合需要多角色协作、粒度更细的协议。
- **CEI（Checks-Effects-Interactions）**：先校验、再更新本地状态、最后做外部交互的经典防重入顺序。
- **Pausable**：紧急暂停机制，用来在发现异常时暂时关闭关键入口，降低损失继续扩散的概率。
- **Timelock**：把高权限操作延迟执行的机制，给社区、监控系统和用户留下反应窗口。
- **tx.origin**：整条调用链最初的 EOA 地址。把它用于权限判断通常很危险，因为中间合约可以代替用户触发逻辑。
- **Signature Replay**：同一份签名被重复利用的攻击。常见防线是 nonce、domain separation 和有效期控制。
- **Delegatecall Risk**：通过 delegatecall 执行外部代码时，状态、权限和升级面会一起暴露出来的风险。
- **Oracle Manipulation**：攻击者通过扭曲价格源或观察窗口，诱导协议做出错误定价或清算判断。
- **Flash Loan**：无需抵押、但必须在同一笔交易内归还的流动性工具。它本身不是漏洞，却会放大攻击者的瞬时资本能力。
- **DoS / Griefing**：不一定直接盗走资产，但能让系统无法执行、成本极高或长期卡死的攻击方式。

## 链与底层网络

- **Account Model**：以账户余额和状态为中心的链上记账模型。以太坊不是 UTXO，而是账户状态机。
- **Chain ID**：链的唯一标识，用来区分主网、测试网和其他 EVM 网络，也是防止跨链签名重放的重要字段。
- **Block**：验证者在某个时间窗口内打包的一组交易与执行结果，是状态推进的基本批次。
- **Block Header**：区块头部摘要，记录父块、状态根、收据根、时间戳等核心元信息。
- **State Root**：当前全局状态的根哈希，代表“这一块执行完以后，整条链的世界状态长什么样”。
- **Receipt Root**：区块内全部交易回执的根哈希，用来证明某笔交易的执行结果被正式纳入区块。
- **Merkle Tree**：把大量数据组织成可验证根哈希的树结构，方便做高效证明。
- **Trie**：以太坊状态、存储和收据使用的前缀树结构，适合做可验证的键值状态组织。
- **Validator**：负责提议和确认区块的参与者，在 PoS 里通过质押和惩罚机制维护网络安全。
- **Finality**：一笔交易被回滚概率降到极低、可以视为“基本定案”的状态。
- **Reorg**：链头发生重组，原先看到的区块被另一条更重或更优的链替换。
- **Fork**：链状态或规则从某个点开始分叉。它既可能是技术升级，也可能是开发测试手段。
- **RPC**：Remote Procedure Call，前端和脚本与节点交互的标准入口。你看到的大多数 `eth_call`、`eth_sendRawTransaction` 都是 RPC。
- **Client**：实现以太坊协议的软件，如执行客户端、共识客户端。它们不是“区块链本身”，而是对协议规则的实现。

## DeFi 与协议机制

- **AMM**：Automated Market Maker，自动做市协议，用资金池和定价规则代替传统订单簿撮合。
- **LP（Liquidity Provider）**：向池子提供资产以换取手续费或激励收益的流动性提供者。
- **k 不变量（k Invariant）**：像 Uniswap V2 这类常数乘积 AMM 用来约束状态变化的核心关系。
- **Slippage**：实际成交价格与预期价格之间的偏差，通常来自池子深度有限或交易量过大。
- **Reserve**：协议内部用于记账和定价的储备状态。很多 AMM 和借贷协议的核心都建立在 reserve accounting 上。
- **TWAP**：Time-Weighted Average Price，时间加权平均价格，用来降低瞬时价格波动对协议判断的干扰。
- **Oracle**：把价格、汇率、利率等外部信息引入链上的机制。没有可靠 oracle，很多 DeFi 协议就无法稳定工作。
- **LTV**：Loan-to-Value，借款相对于抵押品价值的比率，是借贷风险控制的重要阈值。
- **Health Factor**：仓位健康度指标，用来衡量离清算还有多远。它越低，仓位越危险。
- **Liquidation**：当借款仓位不再满足风险约束时，协议强制出售或扣押抵押品的过程。
- **Collateral**：借贷协议里用来担保债务的抵押资产。
- **Stablecoin**：目标价格尽量稳定的链上资产，稳定方式可能来自法币储备、超额抵押或算法调节。
- **Position**：用户在协议中的资产与负债组合，如 LP 仓位、借贷仓位、杠杆仓位。
- **Tick**：Uniswap V3 等协议中用于离散化价格空间的价格刻度单位。
- **Concentrated Liquidity**：把流动性集中部署在特定价格区间内，提高资本效率的设计。
- **Virtual Price**：像 Curve 这类协议里用来衡量池子份额价值变化的派生指标。
- **Restaking**：把已经质押的安全性再次复用给其他服务的机制，目标是把“以太坊安全”扩展给更多系统。
- **AVS**：Actively Validated Service，需要额外验证与惩罚机制来保障正确性的主动验证服务。

## 工程化与训练

- **Foundry**：Solidity 开发工具链，覆盖编译、测试、脚本、追踪和部署，是现代合约工程的高频基础设施。
- **Unit Test**：围绕单个函数或局部行为编写的测试，用来验证基本正确性。
- **Fuzz Test**：通过自动生成大量随机输入去探索边界条件和异常路径的测试方式。
- **Fork Test**：把真实链状态拉到本地再跑测试，用来验证协议在“接近真实世界”的环境里是否仍然成立。
- **Invariant Test**：持续验证某些全局性质在任意操作序列下都不被破坏的测试方式。
- **Coverage**：测试覆盖度指标，用来观察哪些代码被执行过，但它不等于真正的安全性。
- **Deployment Script**：把部署、初始化、配置和校验组织成可重复执行脚本的工程实践。
- **CI**：持续集成流程，把编译、测试、格式检查和静态分析自动化，减少“本地能过、线上翻车”。
- **Runbook**：面向上线、告警、应急或运维的操作手册，帮助团队在高压情况下按步骤行动。
- **Notes Repository**：长期整理源码笔记、问题复盘、图解和结论的知识仓库，是个人成长的可复用资产。
- **Case Study**：围绕某个协议、漏洞、项目或设计问题做的完整案例拆解，用来证明你不仅做过，而且能讲清楚。
- **Artifact**：你能向别人展示的明确学习成果，如项目、笔记、图解、复盘文档、测试仓库。
- **Mock Interview**：模拟面试，通过问答暴露知识断点，让学习从“我以为我会”变成“我真的讲得出来”。
- **Learning Loop**：输入、练习、输出、复盘不断循环的成长机制。真正让人拉开差距的，通常不是单次强度，而是 loop 是否持续运转。
- **Compounding**：前期积累在后续阶段持续放大产出的复利效应。好的学习系统会让旧项目、旧笔记、旧复盘在新阶段继续产生价值。
