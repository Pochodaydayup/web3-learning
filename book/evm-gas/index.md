# Part 3: EVM 与 Gas

## 先理解什么

如果 Solidity 这一章解决的是“合约代码在表达什么”，那么 EVM 和 Gas 这一章要解决的是“这些表达最后在底层怎样被执行、为什么会产生这些成本和限制”。

很多人对 EVM 的第一印象来自一些常见关键词：

- opcode
- stack
- memory
- storage
- calldata
- bytecode
- gas

问题在于，如果这些词只是作为术语存在，它们并不会自然转化成工程能力。  
你可能背下了“storage 贵、memory 便宜、calldata 更省”，但一旦换到真实代码和真实优化选择里，还是会发虚。

所以这一章最重要的不是先记术语，而是先抓住一个核心视角：

> EVM 是一个资源受限的执行环境，而 gas 是对这些资源消耗的计费映射。

只要你先接受这个视角，后面很多现象都会更容易解释。

### 四个最关键的数据区域

对于合约开发者来说，EVM 最需要先抓住的不是所有 opcode，而是四个最关键的区域：

- stack
- memory
- storage
- calldata

可以先这样理解：

- stack：执行时的临时操作栈
- memory：当前调用上下文里的临时可变内存
- storage：合约真正的持久化状态
- calldata：外部调用带入的只读输入

这四者分工不同、成本不同、生命周期不同。  
你后面看到的很多 Solidity 写法，本质上都只是这些分工在语言层的投影。

### 先不要把 Gas 学成“手续费学”

很多人一开始理解 gas，会把它当成一个“链上太贵”的现象。  
但更有用的理解是：

- gas 反映执行和状态操作消耗
- 不同操作成本差异巨大
- 成本差异背后有资源语义

例如：

- 为什么 `SSTORE` 会更贵？
- 为什么 event 常被用来给链下系统提供廉价输出？
- 为什么读取 `calldata` 经常比先复制到 `memory` 更省？

这些都不是随机定价，而是执行模型在成本上的体现。

## 为什么重要

### 你会第一次真正理解“为什么这样写”

当你开始理解 EVM 的资源模型后，许多以前只能死记的写法会变得可解释。

例如：

- 为什么很多函数参数会直接声明为 `calldata`
- 为什么状态结构设计会影响 gas
- 为什么有些地方会选择 `immutable`
- 为什么 slot packing 只在某些场景值得做
- 为什么某些优化能省 gas，但会损伤可读性和可维护性

这会直接改变你阅读别人代码的方式。  
你不再只是看到“这个人好像很会优化”，而是能理解它在优化哪类资源、为了什么、代价是什么。

### 你会更冷静地看待优化

一旦听说“Gas 优化”，很多人会本能地陷入两个极端：

第一种极端是完全不管。  
第二种极端是到处追技巧。

两者都不够成熟。

真正的工程判断应该是：

- 哪些状态写入本来就没必要，先删冗余
- 哪些结构天然更适合当前问题
- 哪些读取路径值得优化
- 哪些优化只是为了省一点点 gas，却严重伤害了代码可读性

对大多数系统来说，最有价值的优化往往不是“黑魔法”，而是：

- 减少无意义状态写入
- 明确数据生命周期
- 避免错误的数据结构选择
- 把链上该做和链下该做的事分清楚

### 你会更容易理解安全和协议设计

Gas 和执行模型不只是性能问题。  
很多安全边界、模块拆分、状态组织方式，本身就和执行成本与状态一致性紧密相关。

例如：

- 为什么某些逻辑不应该放在链上循环里
- 为什么批量操作经常需要特别谨慎
- 为什么某些协议会极度节制状态写入次数
- 为什么复杂系统会把链上与链下职责拆得很清

当你理解 EVM 资源模型时，你也更容易理解真实协议为什么是现在这个样子。

## 从资源视角理解执行

### Stack 不是“一个不常用的底层细节”

很多教程一提 stack 就轻轻带过，好像只是 EVM 的背景知识。  
但它其实很重要，因为 EVM 的许多操作本质上都是围绕栈顶进行的。

你不需要一开始就手写 opcode，但你要知道：

- EVM 执行不是像高级语言那样随意操作对象
- 它是非常机械地围绕栈和内存推进
- 高级语言层的表达最终都会压缩成这种更原始的执行方式

这能帮助你理解为什么有时字节码、低级调用和调试 trace 看起来“很底层”。

### Memory 的价值在于“当前调用内的临时组织”

很多临时计算、拼接和处理，都会落在 memory 里。  
它是可变的，但不会像 storage 一样持久化。

因此 memory 的典型特征是：

- 生命周期仅限于当前执行
- 可用于组织函数内部临时数据
- 相比 storage 更适合作为计算中间层

如果你把它理解成“执行中的工作区”，会比单纯记术语更实用。

### Storage 的本质是链上长期记忆

只要一个值最终要成为合约真实状态，它就会和 storage 打交道。  
这也是为什么 storage 写入常常最昂贵，因为你不是在改一块普通内存，而是在修改链上的持久状态视图。

所以当你设计状态时，应该经常问：

- 这个值真的值得长期存吗？
- 它是不是只服务于链下展示？
- 它是否可以从其他状态推导出来？
- 它是否只是重复信息？

很多合约的 gas 问题，本质上是状态设计问题，而不是“没学会几个优化技巧”。

### Calldata 是非常值得前端开发者理解的区域

因为你每天都在从前端构造参数，所以 `calldata` 对你尤其重要。

可以把它理解成：

- 外部世界传给合约的只读输入
- 编码后的调用数据承载区
- 很多场景下更省，因为它不需要先复制成可变内存

这也是为什么很多只读参数在 Solidity 中更适合直接声明为 `calldata`。

一旦你明白这一点，你看 ABI 编码、函数参数、复杂数组传递时都会更有感觉。

## 常见误区

### 误区一：以为 Gas 优化就是堆技巧

这是最常见的问题。  
很多人对 Gas 优化的理解停留在：

- 变量名短一点
- 到处加 `unchecked`
- 一看见数组就想改
- 一听说 packing 就想硬塞

这些都可能在某些地方有效，但它们不是优化的起点。

真正的起点应该是：

- 这段状态是否必要
- 这次写入是否必要
- 这份数据是否应该落链
- 这段逻辑是否应该放链上执行

如果这些问题都没想清楚，技巧只会掩盖设计问题。

### 误区二：只记“storage 贵、memory 便宜”

这是一个典型的背结论式学习。  
你当然应该知道这个结论，但如果你说不清它为什么成立，那它在真实工程里帮助非常有限。

你至少要知道：

- storage 是持久化状态
- memory 是执行期临时空间
- calldata 是只读输入

只有把资源语义想明白，这个结论才会转化为设计判断。

### 误区三：把 event 当作“便宜版 storage”

event 便宜，是因为它本来就不是拿来替代主状态的。  
它更像是给链下世界看的执行记录。

如果你为了省 gas，把本来需要作为链上主状态存在的信息全塞进 event，短期看起来便宜，长期几乎一定会让系统边界变糟。

### 误区四：看到 slot packing 就想全局套用

slot packing 很值得理解，但不是每个地方都值得为了它牺牲清晰度。

更成熟的做法是：

- 先理解 packing 为什么成立
- 再判断你的合约是否真的受这部分成本影响
- 最后衡量收益是否值得引入复杂度

优化的目标不是“证明你很会”，而是让系统整体更合理。

## 怎么练

这一章最好的学习方式不是单看文章，而是做对比实验。

### 实验一：对比 `storage` 与 `memory`

写两个版本的合约逻辑：

- 一个直接在状态结构上做修改
- 一个先拷贝到临时区域处理再回写

然后对比 gas report。  
不要只看数字，要写清楚：

- 改动了什么
- 为什么数字会不同
- 这种差异说明了什么

### 实验二：观察 slot packing

写几个结构体，尝试不同字段排列顺序：

- `uint256`
- `uint128`
- `bool`
- `address`

观察它们的布局差异。  
做这件事的重点不是为了记住所有规则，而是为了让你第一次意识到“字段排列顺序也会影响底层存储”。

### 实验三：比较 event 与状态写入的使用场景

挑一个业务动作，比如“用户完成签到”或“用户执行一次奖励领取”，思考：

- 哪些信息必须作为状态存下来？
- 哪些信息只需要给链下系统消费？
- 哪些信息完全可以从现有状态推导？

这个练习能让你对“什么应该落链、什么应该作为输出”有更稳的判断。

### 实验四：读一次 gas report

很多人跑过 gas report，但没有真正读懂。  
你的目标不是“看到数字”，而是能说出：

- 哪个函数最贵
- 贵在哪类资源
- 是设计问题还是实现问题
- 有没有值得做的优化

只要你能对着 gas report 讲出这些话，你就已经比只会说“链上好贵”的阶段更进一步了。
