# Part 2: Solidity 与状态

## 先理解什么

很多前端开发者第一次认真学 Solidity 时，最容易把它学成“另一门后端语言”。  
会看见函数、变量、结构体、数组、继承、接口，然后自然地用自己熟悉的语言经验去套它。

但真正重要的地方恰恰不在“它长得像什么”，而在“它最终落到什么上”。

Solidity 的所有设计，归根结底都和三件事强绑定：

- 状态如何持久化
- 执行如何收费
- 权限和失败如何表达

如果你只把 Solidity 当语法学，它会显得不难，但你写出的代码很容易只是能跑。  
如果你把 Solidity 当状态机接口语言学，它的每一个看似普通的语法点都会开始带上工程含义。

例如：

- 为什么 `storage`、`memory`、`calldata` 的差别这么重要？
- 为什么 `mapping` 常常比数组更常见？
- 为什么 event 在合约里到处都是，但又不能拿来替代主状态？
- 为什么错误处理要越来越多地用 custom error？
- 为什么标准实现那么强调权限边界、可见性、继承层次和模块拆分？

这些问题，都不是“语法细节”，而是合约设计的核心。

所以这一章最先要建立的，不是关键字列表，而是一个更稳的理解顺序：

1. 先理解状态放在哪里
2. 再理解不同数据区域如何参与执行
3. 再理解接口、事件、错误这些“对外暴露层”
4. 最后把这些合起来看成一个受约束的链上状态系统

### 先从数据区域开始

在 Solidity 里，你一定会反复见到这三个词：

- `storage`
- `memory`
- `calldata`

它们不是语法噪音，而是 EVM 资源分区在语言层的投影。

可以先粗略理解为：

- `storage` 是持久化状态，合约真正“记住”的东西
- `memory` 是当前执行过程里的临时可变内存
- `calldata` 是外部调用带进来的只读输入

一开始不要急着背所有规则，你只要先记住一个原则：

> 当你写 Solidity，其实一直在决定“这段数据活多久、能不能改、改它要花多大成本”。

### 再看状态结构

对大多数应用合约来说，最常见的状态结构是：

- 基础类型：`uint256`、`address`、`bool`
- `mapping`
- `struct`
- `array`

这里最值得优先建立的认知是：Solidity 的状态结构不是为了“好看”，而是为了适应链上状态读写模型。

为什么 `mapping(address => uint256)` 这么常见？

因为链上最常见的问题，本来就是“给一个 key 查一个 value”。  
比如：

- 一个地址余额是多少
- 一个地址是否已经投票
- 一个 tokenId 的 owner 是谁
- 一个提案当前票数是多少

而 `mapping` 的代价是：它非常适合点查，但天然不适合在链上做完整枚举。  
这就是为什么很多前端开发者会疑惑：“我都存了，为什么拿不全？”

答案是：你存的是一个适合按 key 访问的状态结构，不是一个给链上遍历准备的数据库表。

## 为什么重要

如果你真正理解了状态模型，Solidity 里很多以前只是死记的现象都会变得可以解释。

### 为什么写状态要花钱

看起来只是这样一行：

```solidity
totalSupply += amount;
```

但这并不是普通内存变量加法。  
它意味着你在修改链上持久状态，而链上每一次持久状态写入都会带来真实的资源成本。

这也是为什么：

- 改状态要付 gas
- 纯读取状态通常不需要用户为本地调用付费
- 合约设计要非常在意哪些值真的值得落链

如果你没有这个视角，就很容易写出“功能上合理、状态上冗余、成本上粗糙”的代码。

### 为什么 `view` 不改状态但仍然在“执行”

很多人第一次接触时会误解：

- `view` = 不执行
- transaction = 才执行

其实不是。

`view` 依然会跑代码，只是通常在本地节点环境里做模拟调用：

- 它会读状态
- 会执行逻辑
- 但不会提交状态修改
- 不进入区块共识流程

所以 `view` 的本质不是“没跑”，而是“跑了，但没有以链上提交的方式跑”。

这个区别对你后面理解 gas、失败原因、模拟和真实结果的差异非常重要。

### 为什么 event 很重要但不能当数据库

事件是链下世界理解链上系统的重要窗口。  
前端、索引器、分析系统、通知系统，都会大量依赖 event。

但 event 有一个关键限制：它不是链上主状态。

这意味着：

- 它非常适合链下消费和查询
- 它不适合作为链上业务逻辑的真相来源

如果你把 event 理解成“更便宜的状态”，很容易在系统设计上出错。  
正确的理解应该是：

- state 负责链上真实持久逻辑
- event 负责把执行结果高效暴露给链下世界

### 为什么标准实现看起来那么“啰嗦”

很多刚开始写 Solidity 的工程师会觉得 OpenZeppelin 这些标准实现“层次很多、限制很多、写法很啰嗦”。

但当你开始理解状态和边界以后，你会发现这些看似啰嗦的设计通常都是在解决真实问题：

- 谁能改状态
- 谁能调用关键函数
- 错误如何被一致地表达
- 模块职责如何拆开
- 升级或扩展时如何降低风险

换句话说，标准实现不是为了看起来“像框架”，而是在把常见边界和风险前置编码。

## 状态思维比语法思维更重要

对这本书的目标读者来说，最关键的转折是：不要先问“这段语法怎么写”，而要先问“这个状态系统应该怎么设计”。

例如当你要写一个 Bank 或 Vault 合约时，最先问的问题不应该是：

- `deposit` 怎么写？
- `withdraw` 怎么写？

而应该是：

- 余额记在哪里？
- 谁能存、谁能取？
- 是否有总量概念？
- 是否要记录每次行为？
- 失败应该如何表达？
- 是不是存在重入、越权或精度边界？

只有当这些问题先被想清楚，语法才有正确落点。

这也是为什么很多“看起来语法没问题”的合约，实际上设计非常脆弱。  
因为它们从头到尾都在用语法思维写状态系统。

## 常见误区

### 误区一：把 `public` 当成前端对象的公开属性

在前端或普通后端语言里，“公开”经常意味着“这个字段对外可访问”。  
在 Solidity 里，`public` 更接近于：

- 变量对外可读
- 编译器会自动生成 getter

它不是一个轻描淡写的可见性修饰，而是合约外部接口的一部分。  
一旦你写成 `public`，就意味着你在对链下世界承诺一种读取方式。

### 误区二：把 event 当数据库

很多前端会天然喜欢 event，因为它可监听、可回放、对列表和通知很友好。  
但如果你因此忽视链上状态本体，就会在复杂场景下遇到一致性问题。

event 很适合作为链下订阅与索引入口，  
不适合作为链上业务真实状态的替代品。

### 误区三：把 `memory` 和 `storage` 的差异理解成“只是写法规定”

这类误区通常表现为：

- “这里为什么非要写 `memory`？”
- “编译器让我标这个，照写就行了吧？”

如果你一直这样理解，后面写复杂结构时一定会出问题。  
因为这不是编译器找麻烦，而是在逼你明确：

- 你拿到的是引用还是副本
- 你改的是临时值还是持久状态
- 你的写入有没有真实成本

### 误区四：看到 OpenZeppelin 就只会照抄

标准实现确实应该多用，但“会用标准实现”和“理解标准实现”是两回事。

你至少应该知道：

- `Ownable` 解决什么问题
- `AccessControl` 何时比 `Ownable` 更合适
- `ERC20` 的状态核心是什么
- `ERC721` 和 `ERC20` 在状态结构上最大的不同是什么

否则你只是把别人已经编码好的边界拷了过来，并没有真正建立自己的判断。

## 怎么练

这一章最有效的练习方式，不是刷很多零散语法题，而是按难度渐进地写几个最小合约。

### 练习一：Todo

目标不是做产品，而是练最基础的状态操作：

- `struct`
- `array`
- 状态更新
- 只读与写入函数区分

你需要在写的时候强制回答：

- Todo 存在哪？
- 更新时改了什么状态？
- 为什么某些函数应该是 `view`？

### 练习二：Bank / Vault

这一层开始进入更典型的合约状态思维：

- 用 `mapping(address => uint256)` 记录余额
- 定义存入和提取
- 思考资金流和失败条件

这里会第一次真正把“状态”“权限”“失败路径”三个问题合起来。

### 练习三：Voting

投票合约特别适合练：

- `mapping`
- 权限边界
- 事件表达
- 状态流转

例如：

- 谁能发起提案？
- 谁能投票？
- 同一地址能不能重复投？
- 结果应该通过什么状态读取？
- 哪些地方该发 event？

### 练习四：最小 ERC20

写一个最小 ERC20 非常值得。  
不是为了和 OpenZeppelin 比谁更强，而是为了让你亲手经历：

- `totalSupply`
- `balanceOf`
- `allowance`
- `transfer`
- `approve`
- `transferFrom`

这些机制如何共同构成一个标准 Token 的状态系统。

### 练习时一定要问的四个问题

不管你写哪个练习，都强制问自己这四个问题：

1. 状态存在哪？
2. 谁能改？
3. 谁能读？
4. 失败时怎么暴露原因？

只要这四个问题没有答案，你写出来的代码就还只是“会运行”，还没进入真正的合约设计状态。
