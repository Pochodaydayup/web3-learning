# Part 1: 认知重建

## 先理解什么

如果你以前主要站在前端视角做 Web3，很容易把一次链上交互在心里简化成这样一件事：

> 用户点按钮，前端调钱包，钱包发交易，链上返回结果，前端更新 UI。

这个描述不能说错，但它太粗了。粗到你在遇到真实问题时，几乎没有足够的坐标去判断问题发生在哪一层。

更准确的理解应该是：

1. 前端根据业务动作组织参数
2. 用 ABI 把函数选择器和参数编码成 calldata
3. 钱包根据交易对象做签名
4. 节点接收交易并传播到 mempool
5. 交易被验证者或打包者选入区块
6. EVM 在区块上下文中执行这笔交易
7. 执行过程中可能读取状态、写入状态、发出 event、调用其他合约、回滚
8. 执行结束后生成 receipt、logs 和新的状态结果
9. 前端或索引层再基于这些结果更新界面

只有当你脑子里有这条更完整的链路，很多原本模糊的问题才会开始变清楚。

例如：

- 为什么用户签了名以后交易仍然可能失败？
- 为什么一笔交易可能长时间 pending？
- 为什么 `estimateGas` 并不是绝对可靠？
- 为什么 event 先到了，但你的最终 UI 仍然不能完全信它？
- 为什么同样是“调用合约”，`view` 和真实交易在执行体验上差别那么大？

认知重建的第一步，就是把“链上交互不是普通 API 请求”真正意识到位。

## 为什么重要

很多 Web3 工程上的混乱，不是因为你不知道某个知识点，而是因为你没有一条完整的执行链路作为解释框架。

当没有这条框架时，你看问题会非常碎片化：

- 钱包拒签，看成前端问题
- pending 太久，看成 RPC 不稳定
- 回滚，看成合约“坏了”
- gas 高，看成链“太贵”
- logs 读不到，看成事件监听“抽风”

但实际上，这些现象分别发生在不同层：

- 钱包层
- 网络传播层
- 节点层
- 区块打包层
- EVM 执行层
- 链下索引层
- 前端状态层

一旦你能把问题放回对应层次，定位速度会明显加快。  
这不仅是为了 debug，更是为了让你后面看 Solidity、看 Gas、看安全、看协议源码时，不至于一直把一堆不同层次的概念混在一起。

从工程上说，这一章的重要性在于：它给你后面所有章节提供共同坐标系。

## 从前端动作到链上状态变化

我们拿一个最常见的例子来说明：用户点击“质押”按钮。

表面上你会写出类似这样的前端流程：

```ts
const hash = await walletClient.writeContract({
  address: stakingAddress,
  abi: stakingAbi,
  functionName: "stake",
  args: [amount]
});
```

这行代码背后其实发生了很多层事情。

### 第一层：前端准备参数

前端需要先确定：

- 调哪个合约地址
- 用哪个 ABI
- 调哪个函数
- 参数是什么类型
- 参数是否已经做过单位转换
- 当前链和当前账户是否正确

如果这里出了问题，后面即使交易被正常广播，合约也可能因为参数错误而回滚。

### 第二层：ABI 编码

函数名 `stake(uint256)` 不会直接上链。真正发送出去的是一段编码后的 calldata：

- 前 4 个字节是函数选择器
- 后面按 ABI 规则编码参数

你平时觉得自己在“调用方法”，但链上真正看到的是一段字节数据。  
这也是为什么合约交互和普通 HTTP API 完全不同：EVM 看的不是函数名字符串，而是编码后的调用数据。

### 第三层：钱包签名

交易对象通常包含：

- `to`
- `data`
- `value`
- `nonce`
- `gasLimit`
- `maxFeePerGas`
- `maxPriorityFeePerGas`
- `chainId`

钱包不是在“帮你发请求”，而是在让用户对这份交易内容做签名确认。  
一旦签名完成，这笔交易就成为一份可验证、可广播的链上操作请求。

### 第四层：广播与 mempool

签完名后，交易被发送给节点。节点会先做基础校验，然后把它传播出去，进入待打包交易池，也就是 mempool。

这时交易并没有成功，只是“等待被接受并排序”。  
在这个阶段，交易可能会：

- 因 nonce 问题被拒
- 因 gas 设置不合理被长时间卡住
- 因网络拥堵迟迟不被打包
- 因同 nonce 新交易替换而被覆盖

这就是为什么“用户已经签了名”不等于“链上已经成功执行”。

### 第五层：区块打包与执行

当打包者选中这笔交易后，它才真正进入执行阶段。  
EVM 会在当前链上状态之上执行你的 calldata，对应到合约里的函数逻辑。

这时才会真正发生：

- 状态读取
- 状态写入
- 事件发出
- 内部调用
- 外部调用
- 回滚或成功提交

如果执行中某个 `require`、`revert` 或自定义 `error` 被触发，这笔交易虽然进了区块，也依然会失败。只是失败不是“没发出去”，而是“发出去了，但执行回滚了”。

### 第六层：receipt、logs 与状态

执行结束后，你会得到 receipt。receipt 里通常包括：

- 交易哈希
- 区块号
- gas 使用情况
- 成功或失败状态
- logs

logs 是事件发出的记录，适合链下系统和前端消费。  
但 logs 不是链上状态本身，它只是执行过程的一个输出面。

状态则是 storage 中真正持久化留下来的结果。  
这也是为什么很多时候你不能把“收到了 event”简单等同于“业务状态已经万无一失地可用”。

## 常见误区

### 误区一：把链上交互当成普通 API 请求

这是最常见的误区。  
HTTP 请求通常是：

- 请求发送
- 服务端处理
- 返回结果

链上交互则是：

- 构造交易
- 用户签名
- 网络广播
- mempool 排队
- 区块打包
- EVM 执行
- receipt 返回
- 前端或索引系统再消费结果

两者结构完全不同。

### 误区二：把 `view` 调用和真实交易混为一谈

`view` 调用确实也会跑合约代码，但通常是在本地节点做模拟执行：

- 不消耗真实链上 gas
- 不改链上状态
- 不进入 mempool
- 不需要打包进块

真实交易则完全不同，它要进入链上共识下的执行流程。

### 误区三：以为交易发出去就等于成功

发出去只是进入流程，不是流程结束。  
交易可能会：

- 被节点拒绝
- 一直 pending
- 被替换
- 执行回滚
- 成功执行但前端消费逻辑有问题

所以在前端设计上，“签名成功”“广播成功”“打包成功”“执行成功”应该是不同状态。

### 误区四：把 event 当作链上真相

event 很重要，但它的主要价值是给链下世界提供可检索的执行记录。  
真正的链上真相仍然是状态本身。  
很多系统设计错误，恰恰就出在把 event 误当作主状态来源。

## 怎么练

认知重建最好的方式不是再读十篇概念文，而是做有约束的输出。

### 练习一：把一次真实业务动作拆成九步

挑一个你熟悉的功能，比如：

- 领取奖励
- 质押
- 购买 NFT
- 兑换 Token

把它强制拆成这些步骤：

1. 用户触发了什么
2. 前端准备了什么参数
3. 编码后的 calldata 大致是什么
4. 钱包签了什么
5. 交易进入哪条链的 mempool
6. 打包前会受哪些条件影响
7. 合约执行时会读写哪些状态
8. 成功后会产生哪些 logs
9. 前端最终基于什么更新 UI

如果你能稳定写出这九步，你对系统的理解会清晰很多。

### 练习二：手动跟踪一笔测试网交易

自己发一笔测试网交易，然后记录：

- hash
- nonce
- gas price / fee
- status
- logs
- block number

再用区块浏览器把它完整追一遍。  
只要你认真做过几次，你对“链上到底发生了什么”的感知会强很多。

### 练习三：区分四种前端状态

给你自己的 dApp 交互页做一次状态拆分：

- 钱包弹窗前
- 用户签名后但未打包
- 已打包但执行失败
- 执行成功并完成前端刷新

很多糟糕的 Web3 UX，根源就在于把这些状态全部混成一个“loading”。
