# 从哪些模块开始读最有效

## 先理解什么

OpenZeppelin 并不是拿来“一次性读完整仓库”的。更高效的方式，是把它当作协议阅读训练营，从最能代表核心边界的模块开始。

最推荐的起步顺序通常是：

1. `Ownable`
2. `AccessControl`
3. `ERC20`
4. `ERC721`
5. `Pausable`

这个顺序的好处是：你先掌握权限和标准接口，再去看更复杂模块时，不会被基础边界分散注意力。

### 先把几个词钉牢

**标准（Standard）** 是生态已经形成共同预期的接口、事件和语义规范。直觉上它像你读协议前先掌握的公共语言。工程上这意味着读 OpenZeppelin 或标准代币时，先分清哪些是通用骨架，哪些才是协议自己加的东西。

**继承树（Inheritance Tree）** 是合约通过多层继承形成的结构关系图。直觉上它像一张家谱，告诉你今天看到的行为到底是从哪一代传下来的。工程上这意味着读继承型代码时，如果不先画 inheritance tree，很容易在 override 链里迷路。

**Hook** 是标准流程关键节点预留给子合约扩展逻辑的插槽。直觉上它像主流程里专门留出的可插拔接口。工程上这意味着很多标准库设计的灵活性和风险面，都藏在 hook 附近。

## 为什么重要

很多开发者一上来就读复杂协议时，会同时撞上：

- 不熟悉的状态设计
- 不熟悉的权限模型
- 不熟悉的事件与错误表达
- 不熟悉的继承层次

而 OpenZeppelin 恰好可以把这些“通用工程边界”拆开练。  
你越早把它们读熟，后面看真实协议就越容易把注意力放在业务机制上。

## 核心机制

### 1. `Ownable` 是最小权限模型

读 `Ownable` 时，你重点不是记实现细节，而是问：

- owner 状态是什么
- 关键权限入口在哪里
- 权限转移如何表达
- 失败如何暴露

这会帮你形成“权限模块怎么读”的第一个模板。

### 2. `ERC20` 是标准状态与事件模型的最佳起点

读 `ERC20` 时，重点看：

- `balances`
- `allowances`
- `totalSupply`
- `transfer / approve / transferFrom`
- `Transfer / Approval` 事件

这是很多真实协议都会复用或围绕展开的基础语言。

### 3. `AccessControl` 帮你从“单 owner”走向角色系统

很多真实协议不是一把总钥匙就够，而是需要多角色治理。  
读 `AccessControl` 的价值，就是让你提前习惯：

- 角色标识
- 授权入口
- 权限管理边界

### 4. 阅读 OpenZeppelin 时，重点是“通用边界模板”

你不应该把它当成“别人帮我写好的轮子，我会 import 就行”。  
更重要的是把它读成：

- 标准状态模板
- 标准权限模板
- 标准事件模板
- 标准失败路径模板

这样后面看真实协议时，你就能更快分辨哪些是通用框架，哪些是协议特有设计。

## 常见误区

### 误区一：只会用，不会读

这会导致你越来越依赖标准库，却越来越解释不清标准库到底帮你做了什么。

### 误区二：从最复杂模块开始

这通常会让你同时被继承、抽象和业务细节压住。

### 误区三：读源码只记函数，不记状态和事件

这样你很难真正建立模块结构感。

## 工程判断

以后读 OpenZeppelin 模块时，固定回答这几个问题：

1. 核心状态是什么？
2. 核心外部入口是什么？
3. 核心事件是什么？
4. 核心权限边界是什么？

这四问用熟以后，你读其他协议也会更顺。

## 本节小结

OpenZeppelin 最适合拿来训练“标准边界识别能力”。先把它读成模板库，再去读真实协议，你的阅读负担会显著下降。
