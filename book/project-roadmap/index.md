# Part 7: 项目实战路线

## 先理解什么

很多人都知道“要靠项目练”。  
但真正决定成长速度的，不只是有没有做项目，而是项目顺序对不对。

项目顺序一旦错了，最常见的结果有两种：

- 太简单，做完没有真正进入状态设计和系统判断
- 太复杂，只能一路复制粘贴，最后自己也解释不清

所以这部分不是单纯给你列一串项目名称，而是给你一条更像课程路线的顺序。  
目标不是“做得多”，而是“每做一个项目都新增一层稳定能力”。

推荐顺序如下：

1. ERC20
2. NFT
3. Voting
4. MultiSig
5. Staking
6. DEX
7. Marketplace
8. Lending

如果你一路做下来，几乎会把前面章节的核心能力都压实一遍。

## 为什么重要

### 项目是把抽象理解压回手上的最好方式

你可以读很多理论，也可以读很多源码。  
但只要没有自己做过一轮状态设计、权限边界、失败路径和测试，你的理解就很容易停留在“看着懂”。

项目练习最有价值的地方在于，它会逼你做选择：

- 状态怎么存
- 接口怎么定义
- 权限怎么拆
- event 怎么发
- 错误怎么暴露
- 哪些路径要测

这些选择一旦需要你亲手做，你就会真正感觉到“设计”和“抄实现”之间的差别。

### 顺序合理，学习是叠加的

正确的项目顺序能让你每次只新增一层复杂度。

例如：

- ERC20 让你先理解余额、授权和状态变更
- NFT 让你理解资产唯一性和枚举问题
- Voting 让你理解权限和状态流转
- MultiSig 让你理解签名和执行边界
- Staking 让你理解奖励结算和时间维度
- DEX 让你理解池子、储备量和资金流
- Lending 让你理解抵押、清算和风险控制

如果你一上来就做 Lending，前面这些能力会同时压过来，学习会很痛苦。

## 每个项目真正应该学什么

### 1. ERC20

ERC20 的价值不是“写一个代币”，而是让你第一次真正面对：

- 余额状态
- 总量状态
- 授权模型
- 事件定义
- 转账失败路径

这是绝大多数后续系统都会依赖的基本能力。

### 2. NFT

NFT 的重点不是“会 mint 图片”，而是理解：

- 唯一资产和可替代资产在状态模型上的差异
- `ownerOf`、`balanceOf` 这类关系如何组织
- metadata 该链上放什么、链下放什么

它会让你开始感受到“状态组织方式会随着业务对象变化”。

### 3. Voting

Voting 很适合练：

- 提案状态
- 投票资格
- 一次性约束
- 结果结算

这类项目特别容易暴露你对状态流转的理解是否清晰。

### 4. MultiSig

MultiSig 非常值得做。  
因为它几乎天然要求你同时思考：

- 权限
- 签名
- 执行条件
- 失败路径
- 资金边界

它是很多工程师真正建立“安全感”的一个关键项目。

### 5. Staking

Staking 项目会让你第一次认真处理：

- 时间相关状态
- 奖励累计
- 多用户份额
- 提取和领取的边界

这类逻辑特别适合用来训练你对状态一致性和测试的重视程度。

### 6. DEX

DEX 是一个很明显的复杂度跃迁点。  
它开始要求你把：

- 资产池状态
- 储备量关系
- 定价逻辑
- 资金流
- 事件

放进一个统一系统里理解。

如果你在前面的项目里已经建立了良好习惯，做 DEX 会非常有帮助。

### 7. Marketplace

Marketplace 非常适合训练：

- 订单状态
- 授权和撮合
- 资产与支付分离
- 事件驱动的链下展示

它会把你前面学过的资产、权限、签名、支付逻辑再次整合起来。

### 8. Lending

Lending 建议放在更后面。  
不是因为它“不重要”，而是因为它会把很多前面能力同时叠加：

- 抵押状态
- 利率或参数管理
- 价格依赖
- 健康度判断
- 清算路径

如果前面的基础不稳，这一层会非常容易做成“照着抄一个复杂 demo”。

## 常见误区

### 误区一：一开始就做桥或复杂借贷协议

这是最容易把人劝退的路径。  
因为这些项目不是不能学，而是它们同时需要太多前置能力。

### 误区二：做项目只重功能，不重测试

如果你每个项目都只是把 happy path 跑通，那你学到的会非常有限。  
真正拉开差距的是：

- 你有没有写失败路径
- 你有没有写权限测试
- 你有没有思考状态边界

### 误区三：项目全靠搜模板

模板当然能帮你节约时间，但如果你从头到尾都只是在“改变量名 + 改 UI”，那它不会真正提升你的系统理解。

### 误区四：做完项目不复盘

很多成长发生在“做完之后的解释阶段”。  
如果你做完一个项目，却说不清：

- 它的核心状态是什么
- 最危险的边界是什么
- 你最大的设计失误是什么

那这个项目的收益会打折。

## 怎么练

### 每个项目都写四份材料

为了避免项目沦为“做完就忘”，我建议你每个项目都固定产出四份东西：

1. 状态设计说明
2. 权限模型说明
3. 失败路径清单
4. 测试覆盖说明

这四份东西会强迫你把“代码里隐含的判断”外显出来。

### 用同一套问题复盘每个项目

每做完一个项目，都问自己：

- 这个项目核心状态是什么？
- 哪一段状态设计最容易写错？
- 哪个边界最值得重点测试？
- 最贵的操作是什么？为什么？
- 如果让我再写一次，我会先改哪里？

只要你连续这样做几个项目，成长速度会非常明显。
