# 第 1 章：Solidity 与状态基础

Solidity 的难点从来不只是语法。真正让很多人卡住的，是看起来熟悉的语法背后连接的是陌生的状态模型、陌生的成本模型和陌生的失败边界。本章会把这个基础搭稳。

## 本章要解决什么

本章要解决的是“为什么我会写函数，却不会设计状态”。我们会把 Solidity 重新理解成一种面向链上状态系统的语言，而不是普通后端语言的变体。

你会重点掌握：

- 哪些数据应该进 storage，哪些只该存在于 memory 或 calldata
- 为什么 `mapping` 能高效点查，却不适合链上遍历
- 为什么 `public`、`event`、`error`、`modifier` 都属于接口和边界设计的一部分

## 读完后你应该会什么

- 解释 `storage`、`memory`、`calldata` 的差异以及它们的工程含义
- 设计一个小型合约的核心状态结构
- 分清哪些数据属于主状态，哪些更适合通过 event 暴露给链下
- 看懂基础 OpenZeppelin 模块里大部分状态和权限写法

## 本章核心概念

- **状态变量（State Variable）**：定义在合约级别、决定长期链上状态的数据入口。
- **Storage**：合约持久化保存链上状态的数据区，写入贵但状态能长期存在。
- **Memory**：一次调用执行期间临时使用的数据区，调用结束后就消失。
- **Calldata**：外部调用传入合约的只读输入区，适合读取而不适合修改。
- **Mapping**：适合按 key 点查、不适合天然枚举的链上键值结构。

## 练习与实践入口

最推荐的练法是边读边写。不要先把所有概念背完，再去写合约。你应该在每个小练习里反复问自己：这个值到底应该存在哪里，谁能改它，为什么？
