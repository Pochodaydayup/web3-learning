# 函数边界与对外表达

## 先理解什么

很多人第一次写 Solidity，会天然把注意力放在“逻辑能不能跑通”上。但合约开发里，真正影响可维护性和安全感的，往往不是逻辑本身，而是边界表达是否清楚。

边界表达主要体现在这些地方：

- 函数如何暴露
- 调用者如何被限制
- 结果如何被链下世界消费
- 失败如何被明确表达

这些点看起来分散，实际上是在共同回答一个问题：这个合约打算如何被外部世界使用。

### 先把几个词钉牢

**Visibility** 是 Solidity 用来声明某段能力到底暴露给谁的边界语言。直觉上它像接口上的门禁系统，决定外部用户、继承合约和当前合约自己分别能看到什么。工程上这意味着 `external`、`public`、`internal`、`private` 不是语法装饰，而是接口设计的一部分。

**Event** 是合约对链下世界发出的结构化执行说明。直觉上它像业务完成后发出的广播，方便前端、索引器和分析系统快速知道刚刚发生了什么。工程上这意味着 event 负责可观察性，而不是替代主状态承担最终真相。

**Custom Error** 是用结构化方式表达失败原因的错误机制。直觉上它像给失败路径贴上明确标签，而不是只留下一段模糊字符串。工程上这会直接改善测试断言、前端错误处理和 gas 使用效率。

## 为什么重要

如果边界表达不清晰，常见问题会很快出现：

- 某些本该只供内部复用的逻辑被不必要地暴露成外部接口
- 链下系统不知道哪些行为应该靠 event 追踪，哪些必须回读主状态
- 错误只剩一串字符串，前端和测试很难做结构化判断
- 权限限制写得含糊，导致接口契约不稳定

也就是说，函数、事件和错误不是附属配件，而是合约接口设计的骨架。

## 核心机制

### 1. Visibility 是接口边界的第一层

`public`、`external`、`internal`、`private` 看起来像访问修饰，实际上是在回答“谁能从哪里看到这段能力”。

例如：

```solidity
error ZeroAmount();

event Deposited(address indexed user, uint256 amount);

function deposit() external payable {
    if (msg.value == 0) revert ZeroAmount();
    _recordDeposit(msg.sender, msg.value);
    emit Deposited(msg.sender, msg.value);
}

function _recordDeposit(address user, uint256 amount) internal {
    balances[user] += amount;
}
```

这里 `deposit` 是外部入口，而 `_recordDeposit` 是内部复用逻辑。  
如果一开始不区分这两层，后面接口会越来越混乱。

### 2. Event 是给链下世界的执行说明书

事件的核心价值不是“我也能存点东西”，而是：

- 给前端监听
- 给索引器消费
- 给分析系统回放行为轨迹

成熟的事件设计应该问：

- 哪个行为值得被链下世界观察
- 哪些字段应该 `indexed`
- 哪些信息只是辅助展示，不需要额外落链

你不应该把 event 当数据库，但也不能忽视它在链下可观察性上的价值。

### 3. Custom Error 比字符串更适合工程系统

早期很多教程喜欢写：

```solidity
require(msg.sender == owner, "not owner");
```

而更现代、更工程化的表达通常会倾向于：

```solidity
error NotOwner();

if (msg.sender != owner) revert NotOwner();
```

这样做的收益包括：

- 错误语义更明确
- 测试更容易精确断言
- 前端更容易做结构化处理
- 在很多场景下更节省

### 4. Modifier 的价值在于复用边界规则

`modifier` 不只是语法糖，它在帮助你把重复的边界规则固化成统一入口。

```solidity
modifier onlyOwner() {
    if (msg.sender != owner) revert NotOwner();
    _;
}
```

但成熟使用 modifier 的前提是：你知道它表达的是一致的系统规则，而不是为了让代码“看起来像框架”。

## 常见误区

### 误区一：只关心函数逻辑，不关心接口契约

如果一个函数能跑通，但它的暴露范围、错误表达和事件输出都很混乱，那它仍然是一个设计不成熟的接口。

### 误区二：把 event 当主状态补丁

event 非常重要，但它的主要职责是让链下更好理解执行结果，而不是替代主状态。

### 误区三：把 custom error 理解成“新语法爱好”

它不是为了写法新潮，而是在帮助你把失败路径工程化、结构化。

## 工程判断

写合约接口时，建议始终问这几件事：

1. 这个能力应该被谁看到？
2. 这个行为需要链下监听吗？
3. 失败时我希望前端和测试怎么识别它？
4. 这是不是一个值得复用成 modifier 的规则？

只要这四个问题回答得清楚，很多接口层问题会提前暴露，而不会拖到后期。

## 本节小结

函数决定入口，event 决定链下可观察性，error 决定失败表达，modifier 决定边界规则复用。把它们放在一起看，你才真正是在设计合约接口，而不是只是在堆语法。
