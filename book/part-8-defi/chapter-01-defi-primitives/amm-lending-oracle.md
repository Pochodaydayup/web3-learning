# AMM、Lending 与 Oracle

## 先理解什么

很多人第一次接触 DeFi，会被各种产品名和协议图吓到。其实从开发者角度看，最值得先抓住的不是所有产品细节，而是几个最常见的系统角色：

- AMM：负责资产交换
- Lending：负责抵押、借出和清算
- Oracle：负责给链上带来价格或外部信息

只要先把这三类系统的职责和边界理解清楚，后面的 DeFi 世界会简单很多。

### 先把几个词钉牢

**AMM** 是用池子和定价公式自动完成交易撮合的做市机制。直觉上它像不靠订单簿撮合，而是直接和一个会自己报价格的池子交易。工程上这意味着价格、滑点和流动性都会由池子状态共同决定。

**Lending** 是围绕存款、借款、抵押和清算展开的链上借贷机制。直觉上它像公开、自动化且实时结算的借贷市场。工程上这意味着借贷协议的核心不是“把钱借出去”，而是怎样持续控制风险和维持偿付。

**Oracle** 是把价格或外部信息带进链上协议的输入机制。直觉上它像协议观察外部世界的窗口。工程上这意味着很多 DeFi 风险最终不是公式本身，而是窗口看到的信息是否可靠。

## 为什么重要

DeFi 不是“金融概念上链”这么简单。它真正难的地方在于，链上系统必须在没有中心化结算方的情况下，自行维护：

- 资金流
- 价格关系
- 风险边界
- 清算机制
- 可验证状态

这会直接决定它的合约结构、事件系统、状态设计和安全模型。你理解这些积木，后面读协议代码时才不会只看到表层函数名。

## 核心机制

### 1. AMM 的核心是“用池子状态表达价格关系”

AMM 不依赖传统订单簿撮合，而是通过池子里的资产储备关系来给出价格。最常见的早期模型是恒定乘积思路，你不一定要马上死背公式，但至少要知道：

- 价格不是凭空来的
- 它来自池子当前状态
- 每一次 swap 都在改变池子储备

因此在 DeFi 世界里，价格和状态不是分开的。价格本身就是状态关系的一部分。

### 2. LP 不是“存进去拿点奖励”，而是在承担池子状态角色

流动性提供者向池子提供两侧资产，换来 LP 份额。工程上更重要的理解是：LP 不是简单存款，而是在参与维护池子的可交换性。

这也解释了为什么很多协议需要：

- 记录储备量
- 记录 LP 份额
- 发出 swap、mint、burn 等事件

因为这些状态变化共同构成了池子的行为闭环。

### 3. Lending 协议的中心问题是“如何保证借出的钱仍然安全”

借贷协议不像普通银行那样靠中心化风控，而是靠链上规则约束抵押和清算。最重要的几个概念是：

- 抵押资产价值
- 借出资产价值
- 健康度或抵押率
- 清算阈值

你可以把借贷协议理解成一个持续检查“这笔债还安全不安全”的状态系统。只要价格变化让这份安全边界被打破，系统就必须允许清算者介入。

### 4. Oracle 是 DeFi 的必要输入，也是巨大风险面

很多链上系统没法直接知道“真实价格”。它们需要依赖某种 Oracle 机制来获得价格或外部信息。但外部输入一旦被操纵，整个内部逻辑就可能在完全“按规则执行”的情况下出事。

所以 Oracle 的问题不只是“从哪里拿数据”，更是：

- 数据是否可被瞬时操纵
- 是否只依赖单点来源
- 是否有时间平滑
- 是否和协议决策频率匹配

### 5. MEV 说明链上行为总在博弈环境里发生

DeFi 不是在真空里运行。交易排序、套利者、清算者、搜索者都会影响协议真实表现。这就是为什么很多本来数学上成立的模型，在工程实现中还要考虑：

- 被抢跑
- 被夹子
- 被价格瞬时影响
- 被大资金单笔放大

这也是学习 DeFi 时一定要把执行环境和博弈环境一起带上的原因。

## 工程判断

以后你再看一个 DeFi 协议，先问四件事：

1. 主状态是什么，尤其是资金和份额怎么记录？
2. 价格是怎么来的，是池内状态关系、外部 Oracle，还是两者结合？
3. 风险边界是什么，系统什么时候判定“不安全”？
4. 哪些行为会被外部博弈者放大？

只要这四个问题能答出来，很多复杂协议至少已经有了主干理解。源码阅读时也会更有方向。
