# 前言

## 适合谁读

这本书写给一类很具体的人：已经做过 dApp 前端，但对合约设计和链上底层理解还不够扎实的工程师。

你也许已经做过这些事：

- 接过 MetaMask、Rabby 或 WalletConnect
- 用过 `ethers` 或 `viem` 去读写合约
- 处理过签名、发送交易、等待确认、展示交易状态
- 拉过 event、读过 ERC20 余额、做过 NFT 列表或交易记录
- 踩过网络切换、RPC 超时、用户拒签、pending 卡住、estimateGas 不稳定这些常见坑

从应用开发角度看，这些都已经不算初学者了。问题在于，很多工程师做到这里以后，会进入一种非常典型的“中层停滞”状态：

- 会用，但解释不透
- 会接，但设计不出
- 会查文档，但连不成主线
- 会抄标准实现，但看不懂为什么这么写

这本书就是为这个阶段准备的。

它默认你已经有前端背景，也默认你已经接触过 Web3 的应用层工作。它不会花很多篇幅讲“什么是钱包”“如何调用 `balanceOf`”“怎样配置一个 RPC URL”这种零基础内容。我们会把重心放在更有价值的问题上：

- 一次链上调用到底如何从前端动作变成状态变化？
- Solidity 中最重要的不是语法，而是什么？
- 为什么有些状态结构天然昂贵，有些更适合链下消费？
- 为什么安全和测试在合约开发中不是“后补项”？
- 为什么阅读 OpenZeppelin、Uniswap、Aave 源码会比刷十篇教程更快成长？

如果这些问题是你真正关心的，那么这本书会比一般“Web3 入门教程”更适合你。

## 不适合谁

如果你还没有做过任何 dApp 前端，没有用过钱包签名、没有读写过合约、也不熟悉最基本的 Web3 应用交互，那么这本书可能不是你最省力的第一站。

原因不是门槛高，而是视角不同。

大多数零基础教程会从“如何跑起一个 demo”出发，带你快速建立最初的动手感。这类学习路径非常有价值，因为它会先让你知道整个生态长什么样。但这本书的重点不是让你第一次跑通，而是让你解释得更清楚、设计得更稳、判断得更准。

因此，如果你是下面这类情况，更适合先去做一轮入门项目，再回来看这本书：

- 还没有用过钱包发送链上交易
- 对 ABI、RPC、链 ID 等概念完全陌生
- 还没独立做过任何简单 dApp 页面
- 看到 Solidity 代码会完全不知道它在表达什么

先做一轮入门，再回到这里，你的学习效率反而会更高。因为这本书的很多段落，都是为“踩过一点坑的人”写的。

## 这本书怎么读

这本书最好的读法，不是把它当资讯流一样挑着看，而是先顺着主干路径走一遍，再决定在哪些部分向下深挖。

我建议你按下面这个顺序阅读：

1. 先读“学习地图”，理解整本书的路径设计。
2. 再读“认知重建”，把前端到链上的完整链路在脑中串起来。
3. 接着进入 Solidity、状态、EVM、Gas 这些最核心的技术章节。
4. 然后补安全、工程化与测试。
5. 最后再进入协议源码和项目实战。

为什么不建议一上来就翻最想看的那章？

因为 Web3 的知识点之间高度耦合。你今天看“Gas 优化”，实际上背后连着 storage、state change、opcode、数据布局；你今天看“重入攻击”，背后又连着 external call、状态更新顺序、资金流动和测试策略。如果没有一条稳定主线，你很容易出现“每个词都认识，但连起来就发虚”的状态。

这本书刻意用章节把这些内容压成一条链：

- 从应用层的动作开始
- 到状态和接口的设计
- 到执行环境和成本模型
- 到风险、防守和工程化
- 再到真实协议和项目实践

每一章最后还有“继续深入看什么”，它不是单纯给你一堆链接，而是帮你判断：

- 哪些资料值得先看
- 看它是为了补哪块能力
- 读完之后应该做什么练习

如果你愿意把“读一章 + 做一轮输出”作为节奏，这本书会非常有用。这里的输出可以是：

- 一张流程图
- 一个最小合约
- 一组测试
- 一篇源码阅读笔记
- 一份安全问题清单

只有当你把知识变成解释、图示、代码和判断，它才真正开始成为你的能力。

## 这本书的假设

为了节省你的时间，也为了让我们真正聚焦到核心问题上，这本书会默认一些你已经具备的知识。

默认你已经懂的内容包括：

- 基本前端开发流程
- 常见 JavaScript / TypeScript 项目结构
- 钱包连接和签名的基本使用
- 合约读写、RPC 请求和 ABI 的大致作用
- dApp 页面里的常见状态管理和交互模式

这本书不会反复从零解释这些内容。

相对应地，这本书会重点补上的内容包括：

- 状态为什么这样存、这样改、这样花钱
- 交易为什么会这样执行、这样失败、这样产生回执
- 安全问题为什么会发生，而不是只记住漏洞名称
- 为什么优秀协议源码会这样组织状态、边界和模块
- 为什么合约工程一定要把测试、部署和验证当成系统能力

换句话说，这本书默认你不是“第一次接触 Web3”，而是在寻找一条能把工程认知真正打通的路线。

## 为什么这本书不是从密码学开始

很多人一想到“理解区块链底层”，第一反应就是：

- 要不要先学哈希函数？
- 要不要先补椭圆曲线？
- 要不要先读黄皮书？
- 要不要先理解共识算法？

这些东西重要吗？当然重要。  
但它们是不是你当前成长路径上收益最高的第一步？通常不是。

对已经做过 dApp 前端的人来说，最先需要建立的不是“所有底层理论都懂”，而是“对工程最重要的因果链能讲清楚”。

例如：

- 用户点按钮之后，前端到底构造了什么？
- 钱包到底签了什么？
- 交易进入 mempool 之后发生了什么？
- 合约执行时哪些数据是临时的，哪些会落成状态？
- 为什么有的写法更贵？
- 为什么有的边界有攻击面？

这些问题一旦能解释清楚，你再去看密码学、共识、状态树、数据可用性，理解会快很多。

所以这本书采取的是一条“开发者向下钻”的路线，而不是“从理论最底部往上爬”的路线。它更适合工程师，也更适合已经站在应用层上的你。

## 你最终应该达到什么状态

读完这本书，并认真做掉其中一部分练习后，你不一定会立刻变成顶级协议工程师，但你应该明显进入一个新的层级。

这个层级的表现大致是：

- 你能把链上执行链路讲完整，而不是停留在“发交易就好了”。
- 你能独立设计中小型合约，而不是只能照着模板改名字。
- 你能看懂 OpenZeppelin 这类标准实现的关键设计。
- 你开始具备基础的安全审视能力。
- 你能更有章法地读真实协议源码。
- 你能判断自己下一阶段该学什么，而不是一直在资源海里漂。

这不是“看懂更多名词”，而是工程判断变得稳定。

而这，才是从“会做 dApp 前端”走向“Web3 全栈工程能力”的真正转折点。
