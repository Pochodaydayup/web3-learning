# 从点击到状态变更

## 先理解什么

如果你以前主要从前端视角做 Web3，脑子里很可能有一条过于简化的链路：

1. 用户点击按钮。
2. 钱包弹窗签名。
3. 交易成功或失败。
4. 前端刷新状态。

这条链路不能说错，但它太粗，粗到不足以支持真实工程判断。更准确的理解应该是：

1. 前端根据业务动作组织参数。
2. ABI 把函数与参数编码成 calldata。
3. 钱包把交易对象展示给用户并完成签名。
4. 节点接收交易并传播到 mempool。
5. 打包者根据排序规则选择是否纳入区块。
6. EVM 在当前状态之上执行字节码。
7. 执行过程中可能读取 storage、修改 storage、发出 event、触发内部调用或回滚。
8. 执行结束后产出新的状态、receipt 和 logs。
9. 前端或索引层再消费这些结果，决定什么时候更新界面。

你以后所有关于“为什么会失败”“为什么会贵”“为什么会慢”“为什么和预期不一致”的问题，都应该回到这条链路来解释。

## 为什么重要

这条执行链路的重要性，在于它把很多看似杂乱的链上现象重新归类。

例如“签了名但失败”，可能不是钱包问题，而是执行阶段回滚。  
例如“交易一直 pending”，通常发生在 mempool 和打包排序阶段，而不是 Solidity 代码本身。  
例如“收到了事件但页面数据还不对”，这往往说明你把 event 当成了最终状态，而没有意识到前端还需要等待索引层或再次读取主状态。

一旦你能把问题放回正确层次，排障速度会明显提升。更重要的是，你之后学习 Solidity 和协议设计时，会自然地从“这段逻辑将如何在整条链路里被执行”这个角度思考，而不是只盯着函数本身。

## 核心机制

### 1. 前端并不是在“调用函数”，而是在构造交易输入

以前端最常见的写法为例：

```ts
const hash = await walletClient.writeContract({
  address: stakingAddress,
  abi: stakingAbi,
  functionName: "stake",
  args: [amount]
});
```

这段代码在语言层看像函数调用，但链上真正看到的不是 `stake(amount)` 这个高层表达，而是一个交易对象，其中最关键的是 `to` 和 `data`。

`data` 里包含：

- 前 4 个字节的函数选择器
- 按 ABI 规则编码后的参数

也就是说，EVM 看到的是字节，不是函数名。函数名只是你在前端和 ABI 层用来描述这段字节含义的方式。这也是为什么 ABI、calldata 和 selector 会成为 Web3 工程里反复出现的核心词。

### 2. 钱包签的是交易对象，不是“一个确认动作”

用户在钱包里看到的确认弹窗，本质上是在确认一份交易对象：

```ts
{
  to,
  data,
  value,
  nonce,
  gasLimit,
  maxFeePerGas,
  maxPriorityFeePerGas,
  chainId
}
```

签名的意义，不是“点了同意”，而是用户用私钥对这份内容做了承诺。只要内容被修改，签名就会失效。因此签名前端的职责不是随便把一份交易送进钱包，而是要明确：

- 当前链是不是用户预期那条链
- 目标地址是不是正确合约
- 参数单位是不是已经规范化
- 此时的 nonce、gas 参数是否合理

如果这些前置条件没处理好，签名本身并不能保证业务成功。

### 3. 进入 mempool 只代表“排队”，不代表“成功”

签名完成后，交易被广播给节点。节点会检查签名是否合法、nonce 是否合理、手续费设置是否满足最低传播条件。通过后，交易会被传播到 mempool。

mempool 可以理解成一个待打包交易池，但它不是一个保证队列。交易到了这里，仍然可能：

- 因手续费太低长时间不被打包
- 因同 nonce 的替代交易而被顶掉
- 因账户余额变化而在后续检查中失效
- 因网络拥堵而传播缓慢

所以“交易哈希已经拿到”只代表交易进入了链上流程，不代表结果已经稳定。

### 4. 真正影响业务结果的是 EVM 执行

只有当交易进入区块并被 EVM 执行时，合约代码才真正开始生效。这里会发生几类最重要的动作：

- 读取 storage
- 校验 `require` 条件
- 修改 storage
- 发出 event
- 调用其他合约
- 在失败时回滚全部状态修改

这也是为什么“交易在区块里出现了”不一定等于“交易业务成功了”。如果执行中触发 `revert`，交易会留下一次失败执行记录，但状态修改不会提交。

### 5. Receipt、Logs 与 Storage 是三个不同层次的结果

交易执行后，你通常会看到 receipt，其中包含：

- `status`
- `gasUsed`
- `blockNumber`
- `logs`

很多前端把 `logs` 看得比 `storage` 还重要，因为 event 很容易订阅、过滤和展示。但工程上要分清：

- `storage` 是链上真正持久状态
- `logs` 是执行输出给链下系统消费的记录
- `receipt` 是这次执行的摘要

当三者出现感知差异时，优先以主状态为准，再判断事件和前端数据层是否同步完成。

### 6. `view` 调用只是模拟执行

`view` 和真实交易最本质的差别，不是“一个要钱，一个不要钱”，而是：

- `view` 一般在节点本地模拟执行
- 不进入共识流程
- 不修改最终链上状态
- 不需要打包

因此你在前端 `readContract` 成功，不代表对应业务在真实交易里也一定成功。真实交易会遇到动态状态、gas 上限、打包时序、外部依赖和回滚等一整套额外因素。

## 工程判断

理解了这条链路以后，你在前端和合约协作上要形成几种新的工程判断。

第一，不要把交易状态只做成 `loading/success/error` 三段。更成熟的设计通常至少要区分：

- 等待用户签名
- 已广播，等待上链
- 已打包，等待确认
- 执行成功
- 执行失败

第二，事件监听要服务于“更快反馈”，而不是替代主状态读取。对用户最重要的业务结果，最好仍然有一次来自主状态或索引状态的确认读取。

第三，任何时候只要问题发生，你都先问自己：“这属于参数组织、签名、传播、排序、执行、状态消费中的哪一层？”这个习惯会显著提升你的定位能力。

第四，后面学习 Solidity 时，不要只看语言特性，要不断把每一个特性重新放回这条链路。比如 `event` 属于链下消费接口，`mapping` 属于状态组织，`custom error` 属于失败表达，`calldata` 属于输入承载区。这样你的知识会越学越稳。
