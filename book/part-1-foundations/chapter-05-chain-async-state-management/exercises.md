# 第 5 章练习题

## 概念题

1. 为什么链上前端不适合只用 `loading / success / error` 三段式状态？
2. “链上确认”和“页面数据同步完成”为什么不是一回事？
3. 为什么链上重试不能简单等价于“重新发一次请求”？
4. 为什么说链上前端更接近最终一致，而不是瞬时一致？

## 分析题

1. 某 dApp 在用户签名后立刻显示“交易成功”。请分析这种设计的风险。
2. 如果 indexer 比链上确认慢几个区块，前端应该如何设计过渡体验？
3. 一个交易被 speed up 替换后，前端状态系统最容易出什么问题？

## 代码题

1. 设计一个交易状态枚举，覆盖签名、pending、confirmed、failed、replaced 等阶段。
2. 给一个 staking 页写状态流图，说明哪些阶段依赖钱包、哪些依赖链上 receipt、哪些依赖索引数据。
3. 审查你自己的一个 dApp 页面，找出最可能误导用户的两处状态提示，并写出改进方案。
