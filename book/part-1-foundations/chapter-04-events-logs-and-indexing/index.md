# 第 4 章：Event、Log 与链下索引

到这里，你已经知道一笔交易怎么进入链、怎么花费 gas、怎么成功或回滚。接下来要补上的，是很多前端开发每天都在用、但常常没有真正讲清楚的一层：为什么链下系统总在追事件，为什么事件很好用，却又不能替代真实状态。

## 本章要解决什么

本章要解决的是“我知道合约会 emit event，但不知道 event 在系统里到底承担什么角色，也不知道为什么有些信息适合放 event，有些必须留在状态里”这个问题。你会把 log、receipt、topic、索引器和前端可观测性放进同一张图里理解。

## 读完后你应该会什么

- 说清 event 和链上状态在职责上的根本差异
- 理解 `indexed`、topic、log 查询和 receipt 之间的关系
- 知道为什么很多前端与后端系统依赖事件驱动
- 判断什么时候应该依赖事件，什么时候必须回到合约状态做最终确认

## 本章核心概念

- **Event**：合约主动广播给链下系统消费的结构化执行信息。
- **Log**：事件编码后附着在交易结果里的底层记录形式。
- **Topic**：日志索引字段，帮助链下系统按条件高效过滤事件。
- **`indexed`**：把事件参数放进 topics 的声明方式，用来提升查询便利性。
- **索引器（Indexer）**：消费链上日志并整理成更易查询数据视图的链下系统。
