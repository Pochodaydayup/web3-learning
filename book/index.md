## 这本书适合谁

这本书是写给已经做过 dApp 前端的工程师的。你很可能已经做过钱包连接、签名、交易发送、合约读写、事件监听、列表展示，也已经知道 RPC、ABI、链 ID、gas fee、ERC20、NFT 这些概念在应用层分别扮演什么角色。

但只要你在往下问时出现过这些疑惑，这本书就适合你：

- 为什么同样是“改一个值”，有的操作几乎不疼，有的操作 gas 会明显变高？
- 为什么 `mapping` 这种结构在合约里到处都是，但又总是“拿不全”？
- 为什么很多协议要引入代理、权限控制、暂停开关、签名验证这些看起来麻烦的设计？
- 为什么别人一看代码就知道哪里有重入和权限问题，而自己往往只能凭直觉觉得“不太对”？
- 为什么前端这边明明只是发了一次交易，链上却会产生 receipt、event、状态变化、失败回滚、pending 卡住等一堆不同层次的现象？

如果这些问题你都问过自己，那么你缺的往往不是更多“如何接钱包”的教程，而是一个从应用层一路向下的完整解释框架。

## 你会得到什么

这本书的目标不是把你变成密码学研究者，也不是把所有区块链底层都压在第一阶段学完。它要帮你建立的是一条真正能指导工程判断的主干路径。

读完这本书的第一版，你应该能逐渐做到下面这些事：

- 把一次前端合约调用完整讲成“参数编码 -> 签名 -> 交易广播 -> mempool -> 区块打包 -> EVM 执行 -> receipt/log -> 前端状态更新”的全过程。
- 看到 Solidity 代码时，不再只是辨认语法，而是能判断状态放在哪里、谁能改、谁能读、失败会怎么暴露、哪里容易出安全问题。
- 理解 gas 不是“玄学手续费”，而是对 EVM 资源消耗的一种映射，因此能更冷静地看待优化。
- 明白测试在合约世界里的角色，不只是为了保证功能跑通，更是为了保护状态边界、权限模型和失败路径。
- 开始具备阅读真实协议源码的能力，知道阅读顺序、阅读问题和阅读重点，而不是一打开仓库就被复杂度淹没。

更重要的是，这本书不是一组分散文章，而是一条有节奏的主干路线。每一章都不是孤立的知识点，而是会不断回到同一个目标：让你从“会调用合约的前端”成长为“理解系统因果关系、能独立设计和审视合约系统的工程师”。

## 主干学习路径

这本书故意不从黄皮书、共识算法和密码学证明开始。不是因为这些不重要，而是因为对已经做过 dApp 的前端来说，最短的成长路径不是“从理论最深处起步”，而是先把你每天最容易碰到、也最该先吃透的那条工程主线真正打通。

主干路径分成七个部分：

1. 认知重建  
   先把“前端发起一次交互，到链上真正发生了什么”讲清楚。没有这条链路，后面所有关于失败、pending、gas、安全、事件的理解都会飘。

2. Solidity 与状态  
   这是从“会用 ABI 调别人合约”迈向“开始自己设计状态和接口”的关键一步。重点不在语法背诵，而在状态模型、访问控制、错误暴露和链上成本意识。

3. EVM 与 Gas  
   当你开始理解 stack、memory、storage、calldata 的分工，很多以前只是背下来的结论会突然变得可解释。你会第一次真正知道 gas 到底花在什么地方。

4. 安全  
   Web3 不是“写对功能就行”的环境。你要学会从攻击者视角审视权限、外部调用、签名、价格依赖和状态一致性。

5. 工程化与测试  
   合约写作和合约交付是两回事。Foundry、测试、fuzz、fork test、trace、部署和验证，会决定你能不能把代码从 demo 变成可交付系统。

6. 协议源码阅读  
   真正的成长往往发生在读优秀项目源码时。你会开始看见标准实现、协议设计、状态组织和边界控制是如何在真实项目里协同工作的。

7. 项目实战路线  
   最后把认知压回手上，从 ERC20、MultiSig、Staking、DEX 这样的项目里，把“知道”变成“能做”。

## 这本书和一般 Web3 教程有什么不同

大多数 Web3 教程会在两个方向上走向极端。

第一种极端是只讲接入。你会学到怎么接钱包、怎么发交易、怎么调 ERC20、怎么展示 NFT、怎么在前端拿 event，但核心解释往往停在“先这样写就行”。它能帮你把功能拼出来，却很难帮你建立底层判断力。

第二种极端是直接跳进最硬核的底层理论。你会迅速看到密码学、共识算法、黄皮书公式、状态树和复杂执行细节，但由于缺少应用和工程视角的桥接，很容易在早期就把学习变成信息淹没。

这本书走的是第三条路：从工程者的因果链出发，逐层往下。  
它不回避底层，但会先解决“开发中最常碰到、最影响判断”的问题。它也不只给你链接，而是会把每章的深入资料挂在学习主线后面，让你知道：

- 先读什么
- 为什么先读这个
- 读完之后该练什么

## 你该怎么用这本书

最好的用法不是把它当资讯流随便翻，而是按主干顺序读，并且每读一章就做一小轮输出。这个输出不一定是代码，也可以是你自己的讲解、图示、表格或审查笔记。

例如：

- 读完“认知重建”，你可以把某个熟悉功能的完整交易链路画出来。
- 读完“Solidity 与状态”，你可以自己写一个最小的金库或投票合约，并解释它的状态设计。
- 读完“EVM 与 Gas”，你可以做两组 gas 对比实验，而不是只记几个口号。
- 读完“安全”，你可以故意写出一个有缺陷的合约，再写攻击和修复。

这样读，你获得的不是“看过”，而是能真正迁移到项目里的模型。

## 每章会怎么展开

为了让这本书既能顺着读，也能长期拿来当手册用，每个主章节都会用类似的结构展开：

- 先理解什么
- 为什么重要
- 常见误区
- 怎么练
- 继续深入看什么

最后那一部分尤其重要。它不是为了堆资料，而是为了给你一条明确的下一步：

- 必读：最值得先看的官方资料或标准文档
- 进阶：源码、深入文章、课程或研究材料
- 实战：最适合立刻动手的练习方式

如果你把这本书当成主线，把这些资料当成每一章后面的分支，你会比“漫无目的地刷资源”快得多。
