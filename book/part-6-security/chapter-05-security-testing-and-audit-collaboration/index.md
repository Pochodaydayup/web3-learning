# 第 5 章：安全测试与审计协作

到了安全卷的最后一章，重点已经不只是“知道有哪些风险”，而是“如何在工程流程里把这些风险更早暴露出来，并和审计协同工作”。真正成熟的安全能力，最终一定会落回流程、证据和协作。

## 本章要解决什么

本章要解决的是“为什么很多团队明明知道漏洞名词，也会做一些测试，但上线前仍然无法形成可信安全交付”这个问题。你会学习如何把安全测试、自查清单、风险说明和审计协作组织成更完整的交付过程。

## 读完后你应该会什么

- 设计更有目的性的安全测试与预审查流程
- 区分开发测试、攻击复现测试和审计支持材料的职责
- 理解为什么高质量风险交接会显著提高审计效率
- 建立“安全不是最后丢给审计”的工程意识

## 本章核心概念

- **威胁模型（Threat Model）**：系统假设谁会攻击、会怎么攻击、哪些资产最值得保护的风险图。
- **严重度（Severity）**：用来衡量某个问题真实危害级别的风险分级。
- **修复方案（Remediation）**：把已识别风险转化成具体修改动作的治理方案。
- **审计发现（Audit Finding）**：审计过程中被识别出来并记录的问题条目。
