# 签名边界与代理风险

## 先理解什么

很多开发者第一次接触签名安全时，会把注意力全部放在“能不能 recover 出签名人地址”。这当然重要，但远远不够。真正决定签名是否安全的，往往是它绑定了哪些上下文。

同样地，很多开发者第一次看到 `delegatecall`，会把它理解成“低级调用的一种”。这也太轻了。它真正危险的地方在于：你借用的是别人的代码，但保留的是自己的执行上下文和状态。

## 为什么重要

这两个主题之所以值得一起学，是因为它们都和“上下文是否被正确绑定”有关。

- 签名安全关心：一份授权到底承诺了什么边界
- `delegatecall` 安全关心：一段代码到底在谁的上下文里生效

一旦边界不清，攻击者就会利用这种模糊性。

## 核心机制

### 1. 签名重放不是“验签失败”，而是“验签范围太窄”

如果你只验证“这份签名来自谁”，却不验证“它是给什么场景用的”，那这份签名就可能被：

- 在别的链上复用
- 在别的合约里复用
- 在未来时间反复使用
- 在错误动作上滥用

成熟的签名结构通常至少要绑定：

- `chainId`
- `verifyingContract`
- `nonce`
- `deadline`
- `action` 或业务上下文

```solidity
struct PermitAction {
    address user;
    address spender;
    uint256 amount;
    uint256 nonce;
    uint256 deadline;
}
```

### 2. Nonce 和 Deadline 是授权的时间与次数边界

只要授权不是即时链上执行，而是先签后用，你就必须认真对待：

- 这份签名能不能重复使用
- 这份签名能用到什么时候

nonce 让“一次性”成为可能，deadline 让“过期”成为可能。没有这两层，很多授权即使验签通过，也仍然很危险。

### 3. Delegatecall 的危险是“代码与状态归属分离”

`delegatecall` 在代理升级里很常见：

```solidity
(bool ok, bytes memory result) = implementation.delegatecall(msg.data);
```

看似只是执行了 implementation 的逻辑，实际上更关键的是：

- 改的是当前合约的 storage
- 使用的是当前调用上下文
- 错误会直接污染当前系统状态

这就是为什么升级入口、初始化、实现地址管理和布局一致性都如此关键。

### 4. 签名与代理都需要问“这份权力最终落到哪里”

不管是离线签名还是 `delegatecall`，你都应该反复问：

1. 这份能力的使用边界在哪里？
2. 它会不会被换环境重放？
3. 它最终会改到谁的状态？
4. 谁能触发它？

这是比单纯背漏洞名字更稳的审查方式。

## 常见误区

### 误区一：验出地址就等于安全

远远不够。安全的关键在“签名承诺了哪些上下文”。

### 误区二：`delegatecall` 只是更底层，不代表更危险

恰恰相反。它的强大和危险都来自上下文保留。

### 误区三：签名与代理是两个完全无关的话题

它们表面不同，底层都在处理“边界归属”问题。

## 工程判断

以后审查这两类机制时，先看：

1. 授权边界是否完整绑定
2. 是否有 nonce 和 deadline
3. `delegatecall` 目标地址如何管理
4. 初始化和升级入口是否足够收紧

只要这几步先过，很多高风险问题会更早暴露。

## 本节小结

签名安全不是只会验签，代理安全也不是只认识 `delegatecall`。真正成熟的判断，是能清楚回答一份权力授权给了谁、能在什么上下文里被使用、最终会影响谁的状态。
