# 重入与权限控制

## 先理解什么

安全问题最容易被误解成“有一类特别神秘的黑客技巧”。其实大多数高频漏洞都能还原成几个非常朴素的问题：

- 谁能调用？
- 什么时候把控制权交给了外部？
- 外部回来以后，我之前的状态假设还成立吗？
- 这个输入是不是可能被伪造、重放或操纵？

你把问题看回这些边界，安全就不再是神秘学，而是工程纪律。

### 先把几个词钉牢

**Reentrancy** 是外部调用尚未结束时又重新进入原逻辑路径的风险模式。直觉上它像门还没锁好，人已经第二次冲进来。工程上这意味着状态更新顺序、提款模式和外部调用边界必须一起设计。

**Access Control** 是限制谁能调用哪些能力的权限控制体系。直觉上它像系统里不同房间的门禁规则。工程上这意味着权限问题通常不是“有没有 owner”这么简单，而是每条高风险路径是否真的收得足够紧。

**CEI** 是“Checks-Effects-Interactions”这一先检查、再更新状态、最后做外部交互的经典安全顺序。直觉上它像先确认条件、再锁账、最后才和外界说话。工程上这意味着很多重入风险不是某个 API 危险，而是顺序错了。

## 为什么重要

合约系统和普通应用不同，错误代价通常更高。很多时候不是“出了 bug 再修”，而是一次越权、一处重入窗口、一次错误价格输入，就足以造成真实资金损失。

所以安全思维必须前置到设计阶段，而不是审计阶段。你写代码时就要开始问：

- 这里有没有外部调用？
- 外部调用前后的状态顺序安全吗？
- 哪个函数应该只有特定角色能调？
- 我的签名授权是否绑定了链、合约、用途和时效？

## 核心机制

### 1. 重入的本质是“把控制权交出去以后，世界已经变了”

重入不是单纯的“递归调用”。它真正危险的地方在于：你先把控制权交给外部地址，而外部地址可能在你状态尚未更新前，再次进入你的逻辑。

典型错误顺序是：

1. 先向外部地址转账或调用
2. 再更新本地状态

如果攻击者的地址是合约，就可能借这个窗口再次进入。结果是你的合约还以为“余额还在”“这次还没提取过”，从而被重复利用。

因此最重要的不是记名字，而是记住这个判断：只要你把控制权交给了外部，就要默认原有假设可能失效。

### 2. 权限控制的核心是把“谁能做什么”写死在边界里

很多事故不是因为复杂攻击，而是因为非常基础的权限遗漏：

- 管理函数没有保护
- 高危操作没有角色分层
- 升级入口过宽
- 参数修改权限暴露给了不该有的人

这也是为什么 `Ownable` 和 `AccessControl` 不是“框架装饰”，而是治理边界的代码化表达。你要能判断：

- 是不是一个 owner 就够
- 是否需要细分角色
- 哪些函数属于高危操作

### 3. 签名安全不是“能 recover 地址就行”

离线签名、Permit、订单签名、白名单授权，这些场景都离不开签名。但验签安全的关键不只是“签名人是谁”，而是“这份签名到底承诺了什么上下文”。

至少要考虑：

- 属于哪条链
- 属于哪个合约
- 只能用一次还是可重复用
- 有没有过期时间
- 是为哪个动作授权

只要这些边界没放进签名内容，攻击者就可能重放或挪用一份原本合法的签名。

### 4. `delegatecall` 的风险来自“借用别人的代码，但写你自己的状态”

`delegatecall` 常见于代理升级模式。它执行别人的代码，却使用当前合约的 storage 和上下文。风险就在这里：一旦委托目标不安全，你损失的不是对方的状态，而是你自己的状态系统。

所以看到 `delegatecall` 时要天然警惕：

- 目标地址怎么被设置
- 初始化是否可重复
- 存储布局是否一致
- 升级入口是否安全

### 5. Oracle 与 Flash Loan 让“外部世界”不一定可信

很多协议依赖价格、估值或兑换比例输入。只要这些输入能在短时间内被操纵，你的内部逻辑即使本身写得很正确，也可能被利用。

Flash Loan 的危险往往不在“借钱”本身，而在它让攻击者能在一笔交易里短时放大资金规模，从而影响依赖价格或流动性的逻辑。

所以当协议依赖外部市场信息时，要特别问：这份输入稳定吗，可被瞬时操纵吗，有没有时间加权或多源校验？

## 工程判断

以后做安全审查，建议你先按下面顺序扫一遍：

1. 核心状态是什么，哪些状态最危险？
2. 哪些函数能改这些状态，谁能调？
3. 有没有外部调用，调用前后的状态顺序是否合理？
4. 有没有签名授权，签名边界是否完整？
5. 有没有价格或外部依赖，它们是否可信？

这套顺序的好处，是它能让你从系统边界出发，而不是靠记忆随机翻漏洞词条。长期看，这种方式比死背漏洞名更稳定。
