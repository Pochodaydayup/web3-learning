# Part 6: 协议源码阅读

## 先理解什么

很多工程师在“会写一点合约”之后，下一步会自然想到：是不是该开始读源码了？

这是个非常好的判断，但源码阅读也最容易走错方式。  
因为真实协议仓库往往复杂、分层多、文件多、命名也不总是友好。如果没有方法，你会很快遇到几种典型挫败：

- 打开仓库，不知道从哪一层开始
- 看着每个函数都认识，但不知道整套系统在干什么
- 看了很多行代码，却说不清状态、权限和资金流
- 读了半天只在追实现细节，没有建立结构理解

所以源码阅读的第一原则不是“从头读到尾”，而是：

> 先读结构，再读状态，再读边界，最后才是实现细节。

当你把这个顺序立住以后，源码阅读会从“被复杂度淹没”变成“逐步拆系统”。

### 为什么要读源码

源码阅读对 Web3 工程师的成长速度非常关键。  
因为它是把零散知识真正压回真实工程场景的最快方式。

教程会告诉你：

- `mapping` 是什么
- `modifier` 是什么
- `event` 是什么
- `delegatecall` 是什么

但真实协议源码会告诉你：

- 为什么这里必须用 `mapping`
- 为什么这里要多一层权限抽象
- 为什么这个状态不直接暴露
- 为什么这个 event 是这样定义的
- 为什么升级、暂停、校验被拆到这些层级

换句话说，源码阅读让你从“知道概念”转向“理解工程决策”。

## 为什么重要

### 源码是最好的“设计复盘”

优秀协议源码最大的价值，不只是代码本身，而是它把一整套约束下的设计选择静态地摆在你面前。

你可以从中看到：

- 这个协议核心状态有哪些
- 入口函数有哪些
- 外部依赖有哪些
- 权限如何划分
- 事件如何暴露
- 失败路径如何处理
- 升级和治理如何留边界

这些东西在教程里常常会被拆得很散，  
但在源码里它们是一个整体。

### 它会暴露你真正没弄懂的地方

源码阅读有一个很残酷但很有价值的作用：  
它会让你知道自己哪些地方其实只是“看过”，还没有“懂”。

比如：

- 你以为自己懂 allowance，但一读 ERC20 里的相关逻辑就发现边界没想全
- 你以为自己懂升级模式，但一看代理和实现分离就开始混乱
- 你以为自己懂价格逻辑，但一读 DEX 状态组织就意识到自己只是知道公式名字

这并不是坏事。  
它说明源码开始在帮你校准认知。

## 应该按什么顺序读

顺序非常重要。  
如果你一开始就去读最复杂的借贷协议，大概率不是因为你能力不行，而是因为顺序不对。

### 第一阶段：OpenZeppelin 标准实现

最推荐的起点是：

- `ERC20`
- `Ownable`
- `AccessControl`
- `Pausable`

为什么先从这些开始？

因为它们是大量协议都会反复复用的基础能力。  
你先理解这些模块，后面再看真实协议，就不会被太多“通用边界代码”分散注意力。

你在这一阶段要重点看：

- 核心状态变量有哪些
- 核心外部函数有哪些
- 事件如何定义
- 错误与权限如何组织

### 第二阶段：代理与升级

接下来读：

- Proxy
- UUPS
- Initializable

这里的目标不是立刻精通所有升级模式，而是先理解：

- 代码和状态为什么会分离
- 为什么初始化要格外谨慎
- 为什么存储布局一致性这么重要
- 为什么升级入口本身是极高风险面

这一阶段能让你后面看到复杂协议的代理架构时，不至于完全失去上下文。

### 第三阶段：Uniswap V2

Uniswap V2 是非常好的真实协议入门源码。  
它的复杂度已经足够真实，但仍然比很多后来的系统更容易建立结构理解。

你可以重点看：

- Pair 合约的核心状态
- 储备量如何记录
- Swap 为什么要按那种顺序更新
- Mint / Burn / Swap 的状态流
- event 如何暴露关键动作

读 Uniswap V2 的价值在于：  
你会第一次强烈感受到“状态设计、数学关系、资金流和事件系统是如何绑在一起的”。

### 第四阶段：Aave 或更复杂协议

当你能比较稳定地读懂前面几层后，再进入更复杂系统：

- Aave
- Uniswap V3
- 其他借贷或衍生品协议

这里的重点不再是“能不能看懂某个函数”，而是你能不能在复杂系统里继续抓住主干：

- 核心状态
- 核心入口
- 权限与治理
- 外部依赖
- 风险面

## 常见误区

### 误区一：从最复杂的入口开始

很多人一上来会点开最长、名字最吓人的那个文件。  
这通常是错误起点。

正确做法是先找：

- 核心状态变量
- 最关键的外部入口函数
- 哪些模块是公共基础

先看这些，比先硬读最复杂逻辑有效得多。

### 误区二：只盯函数，不看状态

源码阅读里最致命的习惯之一，就是只往下看函数实现。  
因为合约系统本质上是状态机。  
不先理解状态，函数只是漂在空中的操作片段。

任何时候，先问：

- 这个系统的核心状态是什么？
- 哪些函数在读状态？
- 哪些函数在改状态？

### 误区三：忽略事件和权限

很多工程师看源码只盯业务流程，忽略了：

- 谁能调用
- 哪些操作会发什么事件
- 某个边界如何被保护

但这些地方恰恰是协议设计里最值得学习的部分。

### 误区四：想一次读完

这是不现实的。  
源码阅读更像长期训练，而不是一次冲刺。

更好的方法是：

- 一天读几十行到一两百行
- 读完就写一页笔记
- 持续积累结构感

只要你的节奏稳定，理解会越积越深。

## 怎么练

### 练习一：一页纸读法

每读一个模块，都强制输出一页纸，回答下面这些问题：

- 核心状态有哪些
- 核心入口函数有哪些
- 谁能调用
- 状态或资金如何流动
- 关键事件有哪些
- 最值得继续追的 3 个内部函数是什么

如果你写不出这页纸，说明你还没真正读懂。

### 练习二：画状态和函数关系图

对一个模块画最简单的图：

- 中间写状态
- 四周写入口函数
- 用箭头标明哪些函数读、哪些函数写

这会强迫你从“逐行阅读”切换到“结构阅读”。

### 练习三：写“为什么这样设计”

每次读到一个你觉得“写得很怪”的地方，不要急着判断难看，先试着回答：

- 它解决了什么约束？
- 如果不这么写，会出什么问题？
- 这个选择是在权衡什么？

源码阅读最大的成长，往往就发生在这里。
