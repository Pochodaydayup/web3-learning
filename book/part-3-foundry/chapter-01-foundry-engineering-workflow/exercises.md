# 第 1 章练习题

## 概念题

1. 为什么说 Foundry 更像“验证系统”，而不只是几条命令？
2. `src/`、`test/`、`script/` 三个目录分别承担什么职责？
3. 为什么合约开发里“写完再测”通常不是理想节奏？
4. Trace 在调试合约时能提供什么独特价值？

## 分析题

1. 某同学写完合约后，只在前端点了几次按钮就准备上线。请你从工程闭环角度分析这种做法的主要风险。
2. 为什么部署脚本本身也应该被当成项目资产管理，而不是一次性脚本？
3. 如果一个项目只写单元测试、不写脚本、不看 trace，会在后续交付里遇到什么问题？

## 代码题

1. 初始化一个 Foundry 项目，并创建：
   - 一个最小 `Counter.sol`
   - 一个测试文件
   - 一个部署脚本
2. 为 `Counter` 编写至少 3 个测试：
   - 初始值测试
   - 正常递增测试
   - 非法状态或失败路径测试
3. 使用脚本部署 `Counter` 到本地 `anvil`，并记录部署地址和初始化状态。
