# Foundry 工作流闭环

## 先理解什么

很多人刚开始接触 Foundry 时，最先学到的是一些命令：

- `forge build`
- `forge test`
- `forge script`
- `anvil`

但如果这些命令只是零散存在，你还是很难形成真正的工程闭环。更成熟的理解应该是：

1. 用项目结构组织合约、测试和脚本。
2. 用快速编译与自动化测试建立最短反馈回路。
3. 用 trace、日志和断言验证你的假设。
4. 用脚本和 fork 环境模拟真实部署与真实交互。

也就是说，Foundry 不是一组命令，而是一条“写代码 -> 验证假设 -> 修正设计 -> 再验证”的循环。

## 为什么重要

合约开发和普通前端开发最大的不同之一，是错误成本非常高。你不能像页面样式那样“先差不多跑起来，上线后再修”。这会直接决定：你必须有一套极短、极频繁的验证节奏。

Foundry 的价值正在这里。它让你可以在同一个工程里自然地完成：

- 编译检查
- 单元测试
- 失败路径测试
- fuzz
- fork 测试
- 脚本化部署
- trace 调试

当这些动作的门槛足够低，你就更容易养成好的习惯。很多工程师不是不想写测试，而是反馈回路太长；Foundry 则把这件事变得足够顺手。

## 核心机制

### 1. 先建立清晰项目结构

一个典型 Foundry 项目最少会有下面这些目录：

```text
src/        合约源码
test/       测试文件
script/     部署或运维脚本
lib/        依赖
```

这种结构的好处是，它天然把“系统本体”和“验证系统本体的工具”放在同一个上下文里。你不需要再把测试和部署理解成上线前临时补的东西，它们本来就是项目的一部分。

### 2. 测试不是收尾动作，而是主开发动作

在 Foundry 里，一个很成熟的开发节奏通常是：

1. 先写最小状态和接口。
2. 立刻写成功路径测试。
3. 再补失败路径测试。
4. 如果逻辑涉及复杂输入空间，再加 fuzz。
5. 发现设计不顺，就回到合约重构。

这和“先写完所有代码，再统一测一次”完全不同。你应该越来越习惯在 `forge test` 的反馈里推进设计，而不是写完后再检查。

### 3. Trace 是定位问题的重要窗口

很多前端工程师习惯于只看“报错了没有”。但在合约开发里，你要学会看 trace，因为 trace 会告诉你：

- 调用路径经过了哪些函数
- 哪一步触发了 revert
- 外部调用前后发生了什么
- 状态断言为什么不成立

只要你开始把 trace 当作日常工具，而不是偶尔排怪问题的救命稻草，调试能力会快很多。

### 4. Script 让部署和运维可复现

部署脚本常常被误解成“一次性文件”。其实它们非常重要，因为它们把环境参数、部署顺序和后置检查都显式记录了下来。

一个更成熟的脚本思路应该包含：

- 部署哪些合约
- 初始化参数是什么
- 谁拥有关键权限
- 部署后要读取哪些状态做校验

这样你以后无论是本地测试、测试网还是主网部署，都有一致的执行基础。

### 5. Fork 把本地实验拉近真实链环境

虽然这章只做基础引导，但你要尽早建立一种意识：很多系统真正的风险来自真实链状态，而不是本地 mock。

例如：

- 某个 token 行为可能不完全标准
- 某个 router 需要真实池子状态
- 某个协议交互依赖当前主网配置

Foundry 的 fork 能让你在相对低成本下把这些真实上下文接进来。哪怕还没正式进入复杂 fork 测试，你也应该知道这会是后面非常关键的一步。

## 工程判断

如果只给你一个最重要的 Foundry 学习原则，那就是：不要把它学成“命令大全”，要把它学成“验证系统”。

以后每当你写出一个新合约、新状态或新边界，都主动问：

- 我已经有测试证明它的成功路径吗？
- 我已经有测试证明它的失败路径吗？
- 我是否知道它失败时会返回什么信息？
- 我是否能通过脚本稳定地部署和初始化它？

一旦你形成这种节奏，Foundry 就不再只是工具，而会变成你合约工程习惯的骨架。
