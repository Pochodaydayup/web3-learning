# 第 8 章：静态分析、覆盖率与安全工具链

前一章我们已经把依赖管理和可复现环境讲清楚了。接下来要解决的问题是：为什么一个合约项目即使“测试都绿了”，仍然远远谈不上可放心交付。真正成熟的 Foundry 工程，不会只依赖单一测试视角。

## 本章要解决什么

本章要解决的是“为什么测试通过不等于没有风险；为什么覆盖率不是质量本身；为什么静态分析、属性检查和 CI 工具链要协同工作”这个问题。你会把 Foundry 项目从‘能跑测试’升级到‘有多层防线’。

## 读完后你应该会什么

- 区分单元测试、fuzz、invariant、coverage 和静态分析的不同职责
- 理解覆盖率高不等于场景覆盖完整
- 设计更靠谱的合约质量检查流水线
- 在交付前用工具更系统地发现盲点

## 练习与实践入口

这章很适合在你已经写过一批测试之后来读。重点不是“再装几个工具”，而是重新定义“我凭什么相信这个项目够稳”。

## 延伸阅读入口

理解这章后，你的 Foundry 工作流会更接近真实团队协作：不是只会跑 `forge test`，而是会组织一套持续防错系统。
