# 三种测试模式怎样配合

## 先理解什么

很多刚开始用 Foundry 的工程师，会把测试理解成“写几个函数，然后跑 `forge test` 看绿不绿”。这当然是起点，但距离成熟工作流还差很远。

真正有用的视角是：不同测试模式在验证不同层的假设。

- 单元测试：验证明确逻辑分支
- fuzz：验证更大输入空间
- fork：验证真实链环境假设

它们不是互相替代，而是互相补位。

### 先把几个词钉牢

**单元测试（Unit Test）** 单元测试是针对单个模块或单条逻辑路径做验证的最小测试单元。直觉上它像先在局部房间里检查灯能不能亮，而不是一开始就测试整栋楼。工程上这意味着单元测试适合快速定位具体逻辑回归，是合约工程最基础的安全网。

**模糊测试（Fuzz Test）** 模糊测试是让工具自动生成大量输入，去探索边界值和异常路径的测试方式。直觉上它像不断换着角度去撞系统，看它在哪些输入下最容易露出破绽。工程上这意味着 fuzz 很擅长发现你手写样例时想不到的参数组合问题。

**Fork Test** Fork test 是在本地复制某条真实链的状态后，再在这份分叉状态上运行测试。直觉上它像拿主网现场做排练，而不是只在完全虚构的数据上演练。工程上这意味着你能验证真实 token、真实协议和复杂集成路径，但也必须明确区块高度与外部依赖。

## 为什么重要

如果你只写 happy path 单元测试，很多问题会被漏掉：

- 极端输入
- 不标准 token 行为
- 主网真实状态差异
- 你没想到的状态组合

反过来，如果你每件小事都上 fork，又会让反馈回路变慢，降低日常开发效率。  
所以重点不是“哪种测试最好”，而是“哪种风险该由哪种测试覆盖”。

## 核心机制

### 1. 单元测试负责快速确认局部逻辑

单元测试最适合验证：

- 一个函数的成功路径
- 一个函数的失败路径
- 状态是否按预期变化

```solidity
function testDepositIncreasesBalance() public {
    vm.deal(alice, 1 ether);
    vm.prank(alice);
    bank.deposit{value: 1 ether}();
    assertEq(bank.balances(alice), 1 ether);
}
```

它的价值在于足够快、足够直接，所以应该成为你最常用的第一层验证。

### 2. Fuzz 负责帮你发现“你没想到要测的输入”

当逻辑和输入空间稍微复杂一点时，只靠手工枚举测试值就很不稳。  
这时 fuzz 非常有价值，因为它会不断尝试更多输入组合。

典型适合场景：

- 数学运算
- 边界范围
- 状态更新规则
- 参数规模变化

你可以把 fuzz 理解成“放大版边界感知器”。

### 3. Fork 负责验证真实世界上下文

有些问题不是本地 mock 能暴露的，而是必须放进真实链状态里才看得见，例如：

- 某 token 行为不标准
- 某协议 router 有真实依赖
- 某地址权限配置与你想的不一样

这时 fork 的价值就出来了。  
它不是为了替代本地测试，而是为了把你的系统带进更接近现实的上下文。

```solidity
uint256 mainnetFork = vm.createFork(MAINNET_RPC_URL, BLOCK_NUMBER);
vm.selectFork(mainnetFork);
```

### 4. 成熟测试策略是分层，而不是堆砌

更成熟的工作流通常会是：

1. 先用单元测试把局部逻辑跑稳
2. 再对关键参数空间补 fuzz
3. 对真实集成路径补 fork

这样既能保持开发速度，又能逐层提高信心。

## 常见误区

### 误区一：只测成功路径

这会导致你以为系统“能跑”，但不知道它在失败时是否会稳。

### 误区二：知道 fuzz，却从不真正写

很多人知道 fuzz 有用，但总觉得“以后再补”。结果大量边界问题永远没有被工具帮你发现。

### 误区三：完全依赖本地 mock

如果你的系统将来要和真实协议交互，那长期不做 fork 是非常危险的。

## 工程判断

以后设计测试时，你可以先问：

1. 这是逻辑正确性问题，还是上下文真实性问题？
2. 这是我能明确枚举的输入，还是更适合交给 fuzz？
3. 这条路径有没有真实链依赖？

只要这三个问题先问清楚，测试策略通常就不会太乱。

## 本节小结

单元测试负责快，fuzz 负责广，fork 负责真。真正成熟的 Foundry 工程习惯，不是三者只会其中一个，而是知道它们各自该覆盖哪类风险。
