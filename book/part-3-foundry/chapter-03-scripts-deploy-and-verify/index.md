# 第 3 章：Script、部署与验证

很多合约工程师会写测试，但把部署脚本和验证流程当成“上线那天临时拼一下”的工作。这种做法会让很多本来应该稳定、可复现的工程动作变得脆弱。本章会把这件事系统化。

## 本章要解决什么

本章要解决的是“我会部署合约，但没有把部署当成工程资产”。你会学习为什么部署脚本、环境参数和部署后验证本身都应该被纳入日常工程流程。

## 读完后你应该会什么

- 把部署脚本当成项目的一部分来维护
- 设计更可复现的部署参数与环境区分
- 知道部署后应该验证哪些状态
- 理解为什么“部署成功”不等于“系统已经准备好”

## 本章核心概念

- **部署脚本（Deploy Script）**：把部署、初始化和配置动作程序化的脚本入口。
- **Broadcast**：把脚本里构造好的交易真正签名并发送到链上的步骤。
- **源码验证（Verification）**：把合约源码和编译信息提交给浏览器或平台做可验证公开的过程。
- **幂等性（Idempotency）**：同一动作重复执行也不会把系统推入意外状态的性质。
