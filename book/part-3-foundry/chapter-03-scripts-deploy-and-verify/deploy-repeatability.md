# 部署为什么必须可复现

## 先理解什么

从工程角度看，“部署”不是代码写完后的尾声动作，而是系统生命周期的一部分。你需要清楚回答：

- 部署了什么
- 用什么参数部署
- 部署到了哪个环境
- 部署后如何确认状态正确

如果这些信息只存在于某次命令历史或某个人脑子里，这套系统就不够可靠。

## 为什么重要

部署不可复现，会带来很多麻烦：

- 测试网和主网行为不一致
- 初始化参数来源不清楚
- 权限配置容易漏
- 后续升级或修复很难重演

尤其在合约世界里，部署不是“发个版本就能热修”的轻量动作，它常常决定了真实资产和权限边界。

## 核心机制

### 1. Script 的价值不只是“自动少打几条命令”

一个成熟的部署脚本，真正沉淀的是：

- 目标网络配置
- 构造函数或初始化参数
- 部署顺序
- 部署后检查

```solidity
contract DeployToken is Script {
    function run() external {
        vm.startBroadcast();
        MyToken token = new MyToken("My Token", "MTK");
        vm.stopBroadcast();
        console2.log("Token:", address(token));
    }
}
```

这里保存的不只是动作本身，而是部署事实的最小可重复描述。

### 2. 环境区分要显式，而不是靠记忆

成熟脚本会明确区分：

- 本地
- 测试网
- 主网

以及每个环境的：

- RPC
- deployer
- 参数
- 后续校验逻辑

只有显式区分，你后面才能稳定地做演练、测试和正式发布。

### 3. 部署后验证是脚本的一部分

很多人把部署后验证当成“顺手看一下”。更成熟的做法是把它当成标准步骤：

- owner 是否正确
- 初始供应量是否正确
- 关键地址是否配置正确
- 关键开关是否处于预期状态

换句话说，部署后验证不是可选项，而是部署完成定义的一部分。

### 4. 可复现部署会反过来提升团队协作

当部署路径被脚本化并写清楚以后：

- 你更容易交接
- 更容易复盘问题
- 更容易在不同网络保持一致性
- 更容易让别人接手运维

这就是为什么部署脚本本身也应该像合约和测试一样被认真维护。

## 工程判断

以后设计部署流程时，优先问：

1. 这套脚本别人能不能独立跑通？
2. 参数来源是否清楚？
3. 部署后我如何证明关键状态正确？
4. 如果三个月后重来一次，我还能稳定复现吗？

如果这四个问题能答得清楚，你的部署流程通常已经相当成熟。

## 本节小结

脚本化部署的真正价值，不是少敲命令，而是把部署、环境、参数和验证变成可重复、可交接、可复盘的工程资产。
