# 第 8 章：代码自省、EXTCODE 与 SELFDESTRUCT

前面已经讲过执行模型、调用上下文、存储布局、低级调用、字节码、部署路径和预编译。接下来要补上一块很容易在安全文章和底层资料里反复出现的能力：合约如何观察“某个地址上有没有代码、是什么代码、还能不能被当成稳定代码”。

## 本章要解决什么

本章要解决的是“为什么 `account.code.length` 看起来很简单，却经常被误用；为什么代码存在并不等于身份可被可靠分类；为什么 `SELFDESTRUCT` 在现代 Ethereum 里已经不是你想象中的那种‘彻底删除’”这个问题。

## 读完后你应该会什么

- 理解 EVM 中代码自省相关能力的边界
- 分辨 runtime code、创建期代码和地址身份之间的关系
- 识别基于“合约/EOA 判断”的常见误区
- 理解 `SELFDESTRUCT` 在现代语义下的工程影响

## 本章核心概念

- **EXTCODEHASH**：读取某地址代码哈希、用来判断其代码身份特征的操作。
- **代码内省（Introspection）**：通过读取代码大小、哈希或接口特征来观察合约身份的做法。
- **Selfdestruct 语义**：合约自毁相关行为在现代 EVM 规则下的实际效果与限制。
- **信任边界（Trust Boundary）**：系统开始不能再默认对方行为、数据或代码可信的边界位置。
