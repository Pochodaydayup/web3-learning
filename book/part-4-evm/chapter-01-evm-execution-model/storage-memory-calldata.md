# Storage、Memory 与 Calldata

## 先理解什么

如果把 Solidity 看成一种状态表达语言，那么 EVM 就是负责真正执行这些表达的机器。它不是普通运行时，也不是通用虚拟机的直接翻版，而是一个高度受限、资源明确计费的执行环境。

对合约开发者来说，EVM 里最值得优先抓住的不是所有 opcode，而是四个关键区域：

- stack
- memory
- storage
- calldata

只要这四个区域的职责和成本差异真正清楚了，很多原本显得神秘的 Solidity 设计就会自然落位。

### 先把几个词钉牢

**Stack** Stack 是 EVM 执行时以 256 位字为单位操作的后进先出数据栈。直觉上它像一叠只能从最上面放和拿的卡片，opcode 绝大多数都围绕它工作。工程上这意味着很多看似高层的 Solidity 表达，最终都会被编译成围绕 stack 搬运和计算的低级动作。

**Storage** Storage 是合约长期持久化保存状态的链上数据区。直觉上它像系统的永久档案柜，改动会跨交易继续存在。工程上这意味着 storage 读写是 EVM 成本、状态布局和升级安全的核心来源。

**Calldata** Calldata 是外部调用传入合约的只读输入区。直觉上它像用户交给这次调用的一份密封包裹，合约可以按 ABI 拆开读取。工程上这意味着读取 calldata 往往比先复制到 memory 更省成本，也决定了很多外部函数参数为何标注为 `calldata`。

## 为什么重要

很多工程判断其实都建立在 EVM 资源模型之上。

例如：

- 为什么复杂数组参数常常建议使用 `calldata`
- 为什么状态变量的布局会影响 gas
- 为什么只靠 event 不能替代持久状态
- 为什么外部调用前后的状态顺序如此敏感

如果你只停留在“某些写法更省 gas”的经验层，以后遇到复杂协议和复杂优化时，判断会非常不稳。理解 EVM 之后，你看到的就不再只是“代码风格”，而是“底层资源如何被消耗”。

## 核心机制

### 1. Stack 是执行中的临时操作栈

EVM 的很多指令都围绕栈顶操作展开。你不需要一开始就写 opcode，但要知道，底层执行不是像高级语言那样自由操作对象，而是不断把值压栈、出栈、变换。

这意味着什么？

- 高级表达最终会被拆成很多小步
- 调试 trace 时你看到的是一连串非常机械的执行动作
- “看起来很简单”的一行高级语言，底层可能会展开成一段不短的执行过程

这也是为什么理解 trace 能帮助你更清楚地定位问题：你会看到逻辑是如何一步步走到失败点的。

### 2. Memory 是当前调用里的临时可变空间

`memory` 非常适合承载当前调用过程中的中间数据，例如：

- 临时数组
- ABI 编码时的中间组织
- 外部调用返回数据的处理

它的关键点在于：

- 可变
- 生命周期只限当前调用
- 不会在交易结束后保留

所以你可以把 memory 理解成“执行期工作区”。很多内部运算适合在这里完成，因为它不像 storage 那样承担长期状态成本。

### 3. Storage 是合约真正的持久化状态层

只要一个值需要在交易结束后仍然存在，它就会落到 storage 上。也正因为如此，storage 最值得谨慎设计。

一旦你修改了 storage，就意味着：

- 你在改变链上长期状态
- 这会产生更高成本
- 这类写入会影响协议的整体 gas 模型

所以设计合约时最该先问的问题通常不是“能不能存”，而是“值不值得存”。很多 gas 问题并不是技巧问题，而是长期状态设计过于粗糙。

### 4. Calldata 是只读输入承载区

对前端出身的开发者来说，calldata 非常关键。因为你平时组织的 ABI 输入，最终就会进入这里。

它的特征是：

- 来自外部调用
- 只读
- 适合承载原始输入
- 很多场景下比复制到 memory 更省

这也是为什么很多函数参数会显式标注为 `calldata`。它不是语法细节，而是在表达“这份数据只用于输入读取，不应该作为当前调用中的可变工作区”。

### 5. 调用上下文决定了很多边界

EVM 执行总是发生在一个具体上下文里，包括：

- 当前合约地址
- 当前 `msg.sender`
- 当前 `msg.value`
- 当前可见状态

一旦发生外部调用，控制权就会暂时离开当前上下文。也正是因为这个切换，才会引出安全章节里的很多风险，例如重入和对外部依赖的错误假设。

换句话说，EVM 不只是“算出来一个结果”，而是在一系列上下文切换里维护状态、输入和调用边界。

## 工程判断

理解 EVM 后，你在写 Solidity 时应该逐渐形成这些判断：

第一，尽量让只读输入保持在 calldata 语义里，不要为了习惯而随手拷贝到 memory。

第二，把 storage 当成贵而重要的资源。每次要新增一个状态变量，都问自己：它真的必须是主状态吗，还是可以由别的状态推导，或者仅作为 event 暴露给链下？

第三，遇到复杂调用问题时，回到调用上下文而不是只盯着代码行数。很多问题都藏在“谁在调用”“在哪个上下文里执行”“控制权什么时候交给了外部”。

第四，不要试图一口气学完整个 EVM。对工程师最有价值的顺序是：先掌握资源分工，再理解调用语义，最后再逐步接触更底层的 opcode 和字节码分析。
