# Part 5: 工程化与测试

## 先理解什么

从“会写合约”到“能交付合约系统”，中间隔着一整层工程化能力。

很多人学 Solidity 时，把大部分时间都放在语法、标准和漏洞上，这当然没问题。但一旦进入真实开发，你会很快发现：

- 你需要稳定地跑测试
- 你需要验证失败路径
- 你需要对接真实协议环境
- 你需要追执行 trace
- 你需要比较 gas 差异
- 你需要部署、验证和回放问题

也就是说，真正影响交付效率和可靠性的，不只是“会不会写函数”，而是有没有一套短反馈、可验证、可迭代的工程流程。

对这本书的目标读者来说，我最推荐的起点是 Foundry。  
不是因为别的工具不能用，而是因为 Foundry 在合约开发这件事上，把测试、调试、部署和实验压进了很短的回路里，非常适合你快速建立工程习惯。

这一章你需要先抓住一个基本判断：

> 合约开发不是写完代码再验证，而是写代码的过程本身就应该被验证驱动。

## 为什么重要

### 合约错误代价决定了你不能靠“手点测试”

在普通前端项目里，很多人会先写完，再通过页面点一遍主要流程确认没问题。  
这种方式在某些场景下还能勉强兜底。

但合约开发里，这远远不够。  
因为你需要保证的不只是功能能跑，还包括：

- 权限错误会被正确拒绝
- 状态在失败路径下不会被污染
- 资金流在边界条件下仍然正确
- 合约和真实主网协议交互时没有错误假设

这些问题靠“点一下前端看起来没报错”几乎不可能发现。

### 工程化是把认知变成可重复流程

前面几章你学的是模型：

- 状态模型
- 执行模型
- 风险模型

工程化章节的作用，是把这些模型变成可以反复验证的开发动作。

例如：

- 你认为一个权限边界合理，那就写测试去证明未授权时会失败
- 你认为一个函数逻辑安全，那就 fuzz 它的输入空间
- 你认为和某个协议交互没问题，那就 fork 主网环境去验证

这和普通“写几个单元测试就算完”不是一回事。  
它更接近一种纪律：你不靠印象确认正确，而靠验证流程逼近正确。

### Foundry 的价值在于反馈回路够短

对很多工程师来说，Foundry 最大的优势不是“语法很酷”，而是：

- 写测试快
- 跑测试快
- 看 trace 快
- 复现实验快
- 和主网状态对接快

这种快，会显著改变你的开发习惯。  
一旦反馈足够短，你更愿意：

- 先写测试
- 多测失败路径
- 主动做 gas 比较
- 在改逻辑时立刻验证

这会让合约开发从“写完再祈祷”变成“边写边收敛”。

## 工程化最值得优先建立的几种能力

### 单元测试

单元测试依然是基础。  
它最适合验证：

- 单个函数行为
- 成功路径
- 失败路径
- 状态是否如预期变化

但你不能只把单元测试写成“它会成功执行”。  
更重要的是测：

- 不该成功时会不会失败
- 失败时状态是否保持不变
- 权限不对时是否被挡住

这类测试对合约比对普通前端更重要。

### Fuzz

Fuzz 最有价值的地方，在于它能把你从“我想到哪些输入就测哪些输入”，推进到“让工具帮我在更大的输入空间里找边界”。

它特别适合：

- 算术和范围逻辑
- 状态转换
- 输入规模变化
- 某些你容易低估的角落情况

很多工程师知道 fuzz 是什么，但没有真正把它接进日常开发。  
这很可惜，因为它对合约系统的收益很高。

### Fork Test

Fork test 是很多前端转合约工程后最容易忽视、但收益极高的一步。

为什么？

因为你的代码很可能不是孤立运行的。  
它要和真实世界交互：

- 真实 token
- 真实 router
- 真实 liquidity pool
- 真实权限和账户状态

而在这些场景里，很多“本地 mock 看起来没问题”的假设，一到真实主网状态就会暴露。

fork test 的价值就在于：  
它让你在更真实的状态空间里验证自己的判断。

### Trace 与 Gas Report

测试不是只有 pass / fail 两种信息。  
在合约世界里，你还应该习惯看：

- trace
- revert 原因
- gas report

这些信息能帮助你理解：

- 执行过程走到了哪一步
- 错误到底在什么位置触发
- 哪个函数成本异常高
- 某次优化到底有没有意义

如果你把 trace 和 gas report 当成日常工具，而不是“出了怪问题才看”，成长会快很多。

## 常见误区

### 误区一：只测 happy path

这是最常见也最危险的测试习惯。

你可能会测：

- 用户正常存款
- 用户正常提取
- 用户正常转账

但你更应该测：

- 用户余额不足时是否回滚
- 非管理员是否被拒绝
- 重复调用是否有问题
- 输入为 0 或极端值时系统如何处理

在合约世界里，失败路径不是补充内容，而是核心内容。

### 误区二：把测试当成“证明我写对了”

更成熟的测试心态应该是：

> 测试是为了尽量证明我的假设可能错，而不是自我确认它看起来没错。

这会直接影响你的测试质量。  
如果你只想证明自己是对的，测试很容易写成流程展示。  
如果你真正在找错，测试才会开始覆盖边界。

### 误区三：完全不做 fork test

如果你的系统要和真实协议、真实 token 或真实状态交互，那完全不做 fork test 风险很高。

尤其是前端出身的工程师，往往习惯了 mock 一切。  
但链上系统里，很多关键差异恰恰存在于真实环境：

- 某个 token 行为并不标准
- 某个池子状态和你想的不一样
- 某个 router 在某些分支下行为不同

fork test 能显著减少这类“上线后才知道”的问题。

### 误区四：把部署脚本当一次性文件

很多人把部署脚本当“上线当天临时拼一下”的东西。  
这是很可惜的，因为部署脚本本身也是工程资产。

你应该把它和测试一样认真对待：

- 参数是否明确
- 环境是否可复现
- 验证步骤是否清晰
- 部署后是否有检查动作

对合约系统来说，部署不是“最后一步”，而是系统生命周期的一部分。

## 怎么练

### 练习一：从最小 ERC20 开始写测试

目标不是测试标准库，而是逼自己形成测试节奏。  
至少要覆盖：

- 正常转账
- 余额不足回滚
- allowance 更新
- `transferFrom` 的成功与失败

这个练习能让你把“状态变化”和“失败路径”一起看。

### 练习二：给 Vault 写一套完整测试

Vault 非常适合练工程化，因为它同时涉及：

- 余额状态
- 资金流
- 权限或调用顺序
- 潜在安全问题

你可以要求自己至少写：

- 成功路径测试
- 权限测试
- 失败路径测试
- 一个与重入或外部调用顺序相关的测试

### 练习三：第一次使用 fuzz

不要一开始就在复杂协议上用 fuzz。  
先找一个状态机简单但输入空间明显的场景，例如：

- 充值和提现区间
- 投票权重变化
- 奖励结算

把重点放在“它帮我发现了哪些我没手写出来的边界”上。

### 练习四：对接一次 fork test

选一个你熟悉的真实协议，做一次最小 fork test，比如：

- 和主网某个 ERC20 交互
- 模拟一次 router 调用
- 检查某个真实池子的状态读写假设

一旦你做过一次，你就会更清楚什么叫“真实环境验证”。
